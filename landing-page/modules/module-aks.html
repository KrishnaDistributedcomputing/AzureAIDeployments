Azure Kubernetes Service

Managed Kubernetes for AI and enterprise workloads with autoscaling, workload identity, policy guardrails, and production-grade observability.

View Spec Learn More

Overview

Azure Kubernetes Service (AKS) is Microsoft’s managed Kubernetes platform that offloads control plane operations while preserving CNCF-native APIs and tooling. Teams can deploy containerized AI inference services, APIs, event workers, and internal platforms with enterprise controls built in.

This spec deploys a Standard-tier AKS cluster with autoscaling node pools, private API endpoint support, OIDC issuer + workload identity, Azure Policy addon, and Log Analytics integration for monitoring and alerting.

Key Capabilities

  • Managed Control Plane: Kubernetes masters are operated, patched, and scaled by Microsoft
  • Cluster Autoscaler: Node pool scales dynamically based on pending pods and utilization
  • Private Cluster Mode: API server exposed only through private endpoint within your VNet
  • Workload Identity: OIDC-backed federation for pod-to-Azure auth without secrets
  • Azure Policy Addon: Enforce pod security, image provenance, and governance at admission time
  • Observability: Container insights and logs in Log Analytics with KQL and workbooks
  • Standard Load Balancer: Production ingress/egress with zone-aware resilience
  • Rolling Upgrades: Safe node image and Kubernetes minor upgrades with surge control

Acceptance Gates
  • AKS cluster reports Succeeded with private API endpoint and autoscaler enabled on system node pool
  • OIDC issuer and workload identity validated by successful pod token federation to an Azure resource
  • Azure Policy addon enforces baseline policy set (deny privileged pods, require trusted registries)

  • Rolling deployment + rollback test passes with no service downtime and healthy probes throughout
Deployment Workflow
  1. Provision Log Analytics Workspace for AKS diagnostics
  2. Deploy managed cluster with private API endpoint, RBAC, network profile, and autoscaling system node pool
  3. Enable OIDC issuer + workload identity and bind Kubernetes service accounts to Azure identities
  4. Attach Azure Container Registry and configure image pull permissions for cluster identity
  5. Install ingress and cert management stack (App Gateway Ingress or NGINX + cert-manager)
  6. Apply baseline policies and pod security standards
  7. Deploy workload canary, validate autoscaling and rollback behavior

How to Deploy

Prerequisites
  • Azure CLI 2.57+ with Bicep extension and kubelogin
  • Contributor + User Access Administrator on target subscription
  • Azure Container Registry with signed container images
  • Virtual network design approved (subnets for nodes/ingress)
  • Defender for Cloud plan enabled for AKS
Deployment Steps
  1. Clone repo and navigate to landing-page/specs/
  2. Set kubernetesVersion, node sizing, and private/public API mode
  3. Deploy Bicep template with environment parameters
  4. Fetch kubeconfig and run post-deploy bootstrap manifests
  5. Validate workload identity and policy compliance gates
Azure CLI Example
# Deploy AKS baseline
az deployment group create \
  --resource-group rg-aks-prod \
  --template-file azure-aks.bicep \
  --parameters environment=prod kubernetesVersion=1.30 minNodeCount=2 maxNodeCount=6 \
  --query "properties.outputs" -o json

# Get credentials
az aks get-credentials --resource-group rg-aks-prod --name aks-prod --overwrite-existing

FAQ

When should I choose AKS over Container Apps?

Choose AKS when you need full Kubernetes control: custom CRDs, service mesh, advanced scheduling, daemonsets, or platform-team governance at cluster level. Choose Container Apps for faster serverless-style deployment with less platform operations overhead.

What does Workload Identity replace?

Workload Identity replaces pod-managed identities and secrets-in-config patterns. Pods use OIDC federation and short-lived tokens to obtain Azure access tokens, eliminating long-lived credentials and reducing secret sprawl.

How do I control AKS upgrade risk?

Use staged ring deployments: dev then staging then production, run conformance tests after each upgrade, set max surge for node pool upgrades, and keep rollback playbooks tested. Always pin Kubernetes versions in IaC and avoid auto-upgrade in critical windows.

Can AKS be fully private?

Yes. Enable private API server, use private ingress (internal load balancer or private App Gateway), private ACR endpoints, and restricted egress through Azure Firewall/NAT Gateway. Combine with UDRs and DNS private zones for full network isolation.

What are the top cost levers for AKS?

Node VM sizing, autoscaler min/max bounds, workload right-sizing requests/limits, spot pools for non-critical jobs, and log ingestion volume are the main cost drivers. Set data retention policies and disable noisy logs in non-production environments.
