Azure Virtual Networks

Enterprise-grade network foundation with subnets, NSGs, routing, and hybrid connectivity

DS-VNET-001 Deployment Specification: Azure Virtual Network with hub-spoke topology, multi-subnet segmentation, NSGs, UDRs, and DNS strategy.

Service Specification

DS-VNET-001 Parameter Reference
Parameter Type Purpose
location string Azure region for VNet deployment
environment string Environment name (prod, staging, dev)
addressSpace array RFC 1918 CIDR ranges: 10.0.0.0/16, 10.1.0.0/16
subnetConfig object Subnet definitions: AzureFirewallSubnet, appSubnet, containerSubnet, databaseSubnet
dnsServers array Custom DNS: Azure Firewall IP or 168.63.129.16 (Azure DNS)
enableDdoS bool DDoS Protection Standard (optional, $2944/month)

Dependency Wiring

Hub VNet → Azure Firewall

Central hub (10.0.0.0/16) with AzureFirewallSubnet (10.0.1.0/24) for inspection of all spoke traffic via UDRs.

Spoke VNets → Hub VNet Peering

Bi-directional peering with allowGatewayTransit: true, allowForwardedTraffic: true for hub-and-spoke routing.

Subnets → NSGs

Each subnet has associated NSG: appSubnet allows HTTP/HTTPS, containerSubnet allows ingress only from appSubnet, databaseSubnet allows only from appSubnet:3306/5432.

Custom DNS → Private DNS Zones

VNet linked to private DNS zones for PaaS service name resolution (privatelink.blob.core.windows.net, privatelink.database.windows.net).

UDRs → Firewall NVA

Route tables on app/container subnets forward 0.0.0.0/0 and spoke-to-spoke traffic to hub firewall for centralized inspection.

Acceptance Gates

Gate 1: Address Space Non-Overlap

Validate hub address space (10.0.0.0/16) does not overlap with spoke ranges or on-premises networks via ping/tracert from on-premises gateway.

Gate 2: Subnet Isolation Validation

Deploy test VMs in each subnet; verify NSG rules block forbidden east-west flows (e.g., database subnet rejects HTTP from internet-facing app).

Gate 3: Firewall Rule Enforcement

Test spoke-to-spoke traffic through hub firewall; confirm centralized inspection logs all flows and denies unauthorized protocols.

Gate 4: DNS Resolution

Validate private endpoint DNS resolution for PaaS services (e.g., nslookup myblob.blob.core.windows.net resolves to 10.x.x.x, not public IP).

Gate 5: Hybrid Connectivity

If VPN/ExpressRoute deployed: ping on-premises subnet from Azure VM; verify return path via gateway subnet routing.

Deployment How-To

Step-by-Step Deployment
  1. Prepare Parameters:
    $location = "East US" $environment = "prod" $addressSpace = @("10.0.0.0/16")
  2. Build and Deploy via Bicep:
    az bicep build --file azure-vnet.bicep az deployment group create \ --resource-group myResourceGroup \ --template-file azure-vnet.json \ --parameters location=$location environment=$environment
  3. Create NSGs (post-VNet):
    az network nsg create -g myResourceGroup -n app-nsg \ --rules @nsg-rules.json az network vnet subnet update -g myResourceGroup -n app-subnet \ --vnet-name vnet-hub-prod --nsg app-nsg
  4. Link Private DNS Zones:
    az network private-dns link vnet create \ -g myResourceGroup -n vnet-link \ --zone-name privatelink.blob.core.windows.net \ --virtual-network vnet-hub-prod \ --registration-enabled false
  5. Verify Outputs:
    az deployment group show -g myResourceGroup -n \ --query properties.outputs

Use Cases & Architecture Patterns

Hub & Spoke Topology

Central hub VNet (10.0.0.0/16) with peered spokes for each workload. Azure Firewall in hub inspects all inter-spoke and outbound traffic. Scales to 500+ peerings.

Network Segmentation

Multi-tier subnets (frontend, app, data) with distinct NSGs and UDRs. Denies east-west internet exposure, enforces MFA for management access.

Hybrid Connectivity

VNet integrated with on-premises via Site-to-Site VPN or ExpressRoute (private peering at Layer 3). Extends corporate directory (Entra ID) and DNS into cloud.

Private PaaS Access

Private endpoints for Blob, Cosmos DB, Key Vault in dedicated service subnet. Custom DNS zones resolve privatelink.* FQDNs to private IPs; no public internet exposure.

Multi-Region Deployments

VNets in multiple regions peered for active-active or active-passive failover. Low-latency cross-region replication for data and applications.

FAQ

What are the recommended RFC 1918 address spaces?

Plan for growth: use 10.0.0.0/8 (16M IPs) for large enterprises, 172.16.0.0/12 (1M IPs) for mid-size, or 192.168.0.0/16 (64K IPs) for small networks. Avoid overlap with on-premises and avoid default ranges used by common VPNs (e.g., 192.168.1.0/24).

How do I scale beyond single VNet capacity?

Use hub-spoke topology: one central hub VNet (10.0.0.0/16) can peer with 500+ spoke VNets (10.1.0.0/16, 10.2.0.0/16, etc.). Each spoke isolated by NSG but connected via hub firewall for centralized inspection and routing.

NSG vs. NVA: which should I use?

NSGs (Network Security Groups) are stateful L4 firewalls, free, built-in per subnet—use for basic access control. NVAs (Network Virtual Appliances like Palo Alto, Fortinet) provide L7 inspection, threat prevention, and advanced routing—use for security appliance features or multi-tenant platforms.

Can I modify VNet address space after creation?

Yes, Azure allows address space expansion (e.g., add 10.1.0.0/16 to existing 10.0.0.0/16 VNet), but contraction is restricted. Plan upfront to avoid churn. If resources deployed in old space, migration is complex—best to plan carefully initially.

How do Private Endpoints and DNS work together?

Private endpoints create a NIC (10.x.x.x) for a PaaS service (e.g., Blob). Custom DNS zone (privatelink.blob.core.windows.net) resolves the public FQDN to the private IP. VNet clients query the custom zone and get private IP; public DNS queries fail or return public IP (split-horizon DNS).

What is the maximum number of subnets per VNet?

Azure allows up to 3,000 subnets per VNet. Practical limit is lower due to NSG rules (5,000 rules per NSG). Typical hub is 5–10 subnets; spokes vary by workload.

How do I enable end-to-end encryption in VNet?

Use TLS at application layer and IPsec/VPN for network layer. For on-premises hybrid: Site-to-Site VPN encrypts all tunnel traffic. For Azure-to-Azure inter-region: VNet peering traffic travels over Azure backbone (encrypted), or use Private Link for service-to-service encryption.
