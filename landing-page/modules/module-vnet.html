Azure Virtual Network

Network foundation for Azure workloads with segmentation, private routing, hub-spoke connectivity, and controlled ingress and egress boundaries.

View Spec Learn More

Overview

Azure Virtual Network is the core networking service for secure Azure deployments. It provides address space planning, subnet segmentation, traffic policy control, and private connectivity patterns needed to operate platform and application workloads with enterprise-grade governance.

Key Capabilities

  • Address Space Design: RFC 1918-based private IP planning across environments and regions
  • Subnet Segmentation: Workload isolation for app, container, management, firewall, and data tiers
  • Security Enforcement: NSGs, route tables, and centralized traffic inspection patterns
  • Hub-Spoke Topology: Scalable interconnection model for shared services and workload separation
  • Private Service Access: Integration with Private Link and private DNS resolution strategies
  • Hybrid Connectivity: Integration path for VPN or ExpressRoute to on-premises networks

Common Use Cases

Hub-Spoke Enterprise Network

Centralize shared services and traffic inspection while isolating workload VNets by domain.

Private PaaS Connectivity

Expose platform services privately through private endpoints and DNS zone integration.

Multi-Tier Application Segmentation

Separate web, app, and data paths with subnet and route controls for least privilege networking.

Hybrid Network Extension

Extend existing enterprise network controls into Azure with policy-consistent routing.

How to Deploy

Use this sequence to deploy the VNet foundation from specification and validate traffic and segmentation controls.

1. Prerequisites
  • Azure CLI with Bicep support installed and authenticated to target subscription
  • Approved CIDR allocations for hub, spoke, and subnet structure
  • Network governance controls defined for NSG, UDR, and peering behavior
  • Private DNS and dependency services aligned with application requirements
2. Deployment Steps
  1. Create or select resource group for network foundation.
  2. Build and review the VNet Bicep template.
  3. Deploy with approved location and environment parameters.
  4. Validate peering, routing, and subnet connectivity with test workloads.
  5. Confirm DNS resolution and monitoring readiness for operations handoff.
3. Azure CLI Example
az group create \
  --name rg-network-<environment> \
  --location <location>

az bicep build \
  --file specs/azure-vnet.bicep

az deployment group create \
  --resource-group rg-network-<environment> \
  --template-file specs/azure-vnet.bicep \
  --parameters location=<location> environment=<environment>
4. Validation Checklist
  • Outputs for hub and spoke VNet IDs and subnet IDs are returned
  • Subnet segmentation and routing outcomes match approved architecture design
  • Peering paths and transit controls are verified for expected traffic patterns
  • Private DNS and endpoint name resolution succeed from workload subnets
  • Operational visibility is established for route and policy monitoring

FAQ

Can I resize or change VNet address space after deployment?

You can add address ranges, but shrinking or restructuring active spaces is more complex. Plan CIDR strategy up front for long-term growth.

Why use a hub-spoke model for enterprise landing zones?

Hub-spoke centralizes shared controls and services while preserving workload isolation in spokes, improving governance at scale.

How does Private DNS relate to Private Endpoints in VNets?

Private DNS zones map service FQDNs to private endpoint IPs inside the VNet so workloads resolve and connect through private network paths.
