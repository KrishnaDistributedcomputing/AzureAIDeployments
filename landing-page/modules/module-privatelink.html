Azure Private Link

Access Azure PaaS services privately over your virtual network — no internet exposure, full DNS integration, and data exfiltration prevention built in.

View Spec Learn More

Overview

Azure Private Link enables private connectivity to Azure services and customer-hosted services by mapping them to a private endpoint — a network interface with a private IP address inside your VNet. Traffic never traverses the public internet, making it foundational to Zero Trust and compliance-driven architectures.

The spec deploys a Private Endpoint for Azure OpenAI (Cognitive Services), a matching Private DNS Zone (privatelink.openai.azure.com), a VNet link, and a DNS A record pointing to the endpoint's private IP — providing complete, end-to-end private resolution.

Key Capabilities

  • Private Endpoints: Dedicated NIC in your VNet mapped to the target Azure resource
  • Private DNS Integration: Automatic DNS resolution to private IP from within the VNet
  • No Internet Exposure: PaaS services accessed entirely over Microsoft backbone — no public IP required
  • Data Exfiltration Prevention: Policy controls restrict which services can be reached, closing exfiltration channels
  • Cross-Region Support: Connect across Azure regions privately via the Microsoft global network
  • On-Premises Reach: Combine with VPN Gateway or ExpressRoute to reach private endpoints from on-prem
  • Compliance Ready: Satisfies PCI-DSS, HIPAA, ISO 27001, and SOC 2 network isolation requirements
  • Custom DNS Support: Works with custom DNS forwarders for hybrid resolution scenarios

Dependency Wiring

NIC deployed into subnetId with a private IP. Connects to the target PaaS via privateLinkServiceConnections. Custom NIC name: nic-pep-openai.

Name: pep-openai-{environment}

Private DNS Zone

Zone privatelink.openai.azure.com deployed globally. Linked to vnetId with registration disabled. Enables private DNS resolution within the VNet.

registrationEnabled: false

DNS A Record

A record with TTL 3600 pointing to the private endpoint NIC's privateIPAddress. Ensures VNet-internal name resolution routes to the private NIC automatically.

TTL: 3600 seconds

Acceptance Gates
  • Private Endpoint NIC provisioned in target subnet with a private IP from the subnet address range
  • Private DNS Zone privatelink.openai.azure.com linked to the VNet with registrationEnabled: false
  • DNS A record resolves to private endpoint IP from within the VNet (verify with nslookup from a VNet VM)
  • Public DNS for the service still resolves to its public IP from outside the VNet (split-horizon confirmed)
  • Target PaaS service reachable at private IP; public endpoint disabled or firewall-restricted
Deployment Workflow
  1. Identify target PaaS resource ID and its groupId (e.g. account for Cognitive Services, blob for Storage)
  2. Ensure the subnet has privateLinkServiceNetworkPolicies: Disabled and sufficient address space
  3. Deploy Microsoft.Network/privateEndpoints into the subnet, referencing the target service
  4. Deploy Microsoft.Network/privateDnsZones (global location) with the correct zone name for the service
  5. Create a Virtual Network Link to attach the DNS zone to the VNet
  6. Create DNS A record pointing to the private endpoint's NIC private IP
  7. Disable or restrict the public endpoint on the target PaaS service via its own firewall/network rules
  8. Verify outputs: privateEndpointId and privateDnsZoneId recorded; DNS resolution tested

How to Deploy

Prerequisites
  • Azure CLI 2.57+ with Bicep extension
  • Existing VNet and subnet (PE policies disabled)
  • Target PaaS resource deployed and resource ID known
  • Network Contributor + Private DNS Zone Contributor roles
Deployment Steps
  1. Collect vnetId, subnetId, and target PaaS resource ID
  2. Update privateLinkServiceId and groupIds in the Bicep file
  3. Run az deployment group create (see below)
  4. Disable the public endpoint on the target service
  5. Run DNS validation from a VM inside the VNet
Azure CLI Example
# Deploy Private Link resources
az deployment group create \
  --resource-group rg-networking-prod \
  --template-file azure-privatelink.bicep \
  --parameters environment=prod \
               vnetId=/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.Network/virtualNetworks/{vnet} \
               subnetId=/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.Network/virtualNetworks/{vnet}/subnets/{subnet} \
  --query "properties.outputs" -o json

# Verify DNS resolution from within the VNet
nslookup your-resource.openai.azure.com

FAQ

How does DNS resolution work with Private Link?

Within the VNet, the Private DNS Zone overrides public DNS — the service FQDN resolves to the private endpoint's IP. Outside the VNet, public DNS still resolves to the public IP. This split-horizon pattern ensures correct routing without changing application configuration.

Can I access Private Link endpoints from on-premises?

Yes. Connect your on-premises network to the VNet via VPN Gateway or ExpressRoute, then configure a DNS forwarder (e.g. Azure DNS Private Resolver) to forward queries for privatelink.* zones to the Azure Private DNS Zone. Applications on-prem then resolve to the private IP.

What is the groupId and how do I find it for my service?

The groupId identifies the sub-resource of the target service. Common values: account (Cognitive Services / OpenAI), blob / file / queue (Storage), vault (Key Vault), sql (SQL MI). Use az network private-link-resource list to enumerate supported groupIds for any resource.

Is there a limit on private endpoints per subscription?

The default quota is 1,000 private endpoints per subscription. This can be increased by submitting a support request to Microsoft. Plan for one endpoint per service instance per network boundary.

Should I disable the public endpoint after deploying Private Link?

Yes, for a security-compliant deployment. After verifying the private endpoint works, disable or restrict the service's public endpoint via its own network/firewall rules. Leaving both active creates a redundant public exposure path that undermines the Privacy Link security boundary.
