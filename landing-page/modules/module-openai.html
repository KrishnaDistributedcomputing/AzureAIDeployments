Managed access to OpenAI's GPT-4, GPT-4 Turbo, embeddings, and DALL-E models with enterprise security and compliance.

Overview

Azure OpenAI Service provides REST API access to OpenAI's powerful language models, as well as image generation capabilities through DALL-E 3. It offers enterprise-grade security, regional availability, and integration with Azure authentication and monitoring.

Key Capabilities

  • Large Language Models: GPT-4, GPT-4 Turbo, GPT-3.5-turbo with custom fine-tuning
  • Embeddings: text-embedding-3-large and text-embedding-3-small for semantic search
  • Image Generation: DALL-E 3 for image creation and variation
  • Enterprise Security: Network isolation, managed identity, role-based access control
  • Global Deployment: Available across 10+ Azure regions with SLA guarantees
  • Token Rate Limiting: Flexible quota management and consumption tracking

Common Use Cases

🔍 RAG Systems

Build retrieval-augmented generation pipelines with vector embeddings and semantic search.

💬 Chat Applications

Create conversational AI experiences with multi-turn chat and context management.

📝 Content Generation

Automate content creation for documentation, emails, and marketing materials.

🖼️ Image Generation

Generate, edit, and create variations of images using DALL-E 3.

✓ Acceptance Gates

All gates must pass before the deployment is considered production-ready.

  • Gate 1 — Endpoint health: GET /openai/deployments/{id}/completions must return HTTP 200 within 2 seconds at P95 under normal load.
  • Gate 2 — Network isolation: A direct public endpoint call (bypassing Private Link) must return HTTP 403 in any environment where privateEndpointEnabled = true.
  • Gate 3 — Content filter: Submitting a known policy-violating prompt must return finish_reason: content_filter, not a model completion.
  • Gate 4 — Key Vault resolution: The application must retrieve the OpenAI API key from Key Vault at startup. Any hardcoded key in config fails this gate.
  • Gate 5 — Quota compliance: Deployed TPM ceiling must not exceed the approved subscription quota documented in the environment cost approval.
🚀 Spec Workflow
  1. Author: Complete all required parameters. Resolve every <placeholder> field before submission.
  2. Review: Submit spec for peer review. At least one reviewer who is not the author must approve before status advances to Approved.
  3. CI gate: Spec linting and dependency validation run automatically on PR. The IaC template is generated from spec inputs — no manual edits permitted.
  4. Deploy: Approved spec triggers the Infrastructure-as-Code pipeline to provision or update the Azure OpenAI account and model deployments.
  5. Validate: All five acceptance gates must pass in the target environment within 30 minutes of deployment completion.

2. Purpose & Scope

Purpose: Define the deployment specification for Azure OpenAI Service in the production environment, including model selection, quota allocation, network configuration, and security controls required before infrastructure provisioning is requested.

In scope: Azure OpenAI account, GPT-4o and text-embedding-3-large model deployments, Private Endpoint, Key Vault integration, diagnostic logging.

Out of scope: Application-layer prompt engineering, fine-tuning workflows, downstream RAG pipeline configuration.

3. Functional Requirements
  • FR-01 — The service must expose a dedicated GPT-4o deployment slot with a minimum capacity of 40K TPM.
    Rationale: Peak load in production is projected at 35K TPM; 40K provides 14% headroom.
  • FR-02 — The service must expose a text-embedding-3-large deployment slot with a minimum capacity of 25K TPM.
    Rationale: Embedding workloads run concurrently with chat and must not share quota.
  • FR-03 — The service must return HTTP 200 from GET /openai/deployments/{id}/completions within 2 seconds at P95 under normal production load.
    Rationale: SLA commitment to product team.
  • FR-04 — The service must enforce Azure AI Content Safety at level Medium for the hate, sexual, violence, and self-harm categories on all completions endpoints.
    Rationale: EU AI Act compliance — content filter must be on and documented.

5. Security Controls
  • Authentication: Managed Identity only — API key authentication must be disabled at the resource level.
  • Authorization: RBAC using built-in role Cognitive Services OpenAI User; no Owner/Contributor assignments to the OpenAI resource.
  • Key storage: Endpoint URL stored in Key Vault secret openai-endpoint-prod; no keys in app config or environment variables.
  • Network: Private Endpoint in subnet snet-ai-prod; DNS zone privatelink.openai.azure.com linked to hub VNet.
  • TLS: TLS 1.2 minimum enforced at Private Endpoint.
  • Audit logs: All control-plane operations and data-plane request logs routed to Log Analytics workspace law-ai-prod-eastus with 90-day retention.

8. Rollback & Recovery
  1. Model deployment rollback: Delete the failing deployment slot via ARM API; prior slot version remains active. Max recovery time: 5 minutes.
  2. Account rollback: Restore from ARM template snapshot tagged pre-deploy-{timestamp} stored in the deployment pipeline artifact store.
  3. Escalation: If G-02 (network isolation) fails post-deploy, immediately set publicNetworkAccess = Disabled via emergency change request; notify Security within 15 minutes.
