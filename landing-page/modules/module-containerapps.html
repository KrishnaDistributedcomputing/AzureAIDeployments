Azure Container Apps

Serverless container platform for microservices, event-driven apps, and APIs with Dapr integration, zone redundancy, and scale-to-zero built in.

View Spec Learn More

Overview

Azure Container Apps is a fully managed serverless container platform built on Kubernetes and KEDA. It abstracts away infrastructure complexity while delivering the full power of containers including auto-scaling, service discovery, ingress, and Dapr-based distributed application patterns.

Container Apps enables polyglot microservices architectures with zero-downtime traffic splitting, event-driven scaling from KEDA triggers, and deep integration with Azure Monitor through built-in Log Analytics.

Key Capabilities

  • Scale-to-Zero: Containers scale down to zero when idle, eliminating idle costs
  • KEDA Event-Driven Scaling: Scale on HTTP, queues, topics, cron, and custom metrics
  • Dapr Integration: Native support for state, pub/sub, bindings, and service invocation
  • Revision-Based Deployments: Traffic splitting between revisions for blue/green and canary releases
  • Managed Ingress: HTTPS with free TLS certificates and automatic DNS via FQDN
  • Zone Redundancy: Multi-zone environment deployment for high availability
  • Managed Identity: SystemAssigned identity for secure Azure service access without credentials
  • VNET Integration: Private environment for internal-only workloads

Use Cases

Microservices

Deploy polyglot microservices with built-in service discovery, load balancing, and Dapr-based distributed patterns.

Event-Driven Apps

Trigger containers from Service Bus, Event Grid, Storage queues, and Kafka topics via KEDA without always-on compute.

API Backends

Deploy REST and gRPC APIs with external HTTPS ingress, auto-scaling to handle traffic bursts, and zero-downtime deployments.

Background Jobs

Run scheduled or queue-triggered batch jobs without infrastructure management, scaling from zero when work arrives.

Acceptance Gates
  • Container App Environment deployed with zoneRedundant: true and Log Analytics destination configured
  • Container App reachable via FQDN with valid HTTPS certificate (no browser security warnings)
  • SystemAssigned identity active; ACR pull or Key Vault access verified via RBAC role assignment
  • Scale-out to maxReplicas triggered under HTTP load test; scale-in back to minReplicas confirmed after test
  • Container logs visible in Log Analytics workspace via KQL query within 5 minutes of deployment
Deployment Workflow
  1. Provision Log Analytics Workspace (Microsoft.OperationalInsights/workspaces)
  2. Create Container App Environment (Microsoft.App/managedEnvironments) linking Log Analytics keys
  3. Build and push application image to Azure Container Registry
  4. Deploy Container App (Microsoft.App/containerApps) with image reference, ingress, scale rules, and SystemAssigned identity
  5. Assign RBAC roles (AcrPull, Key Vault Secrets User) to the Container App managed identity
  6. Verify outputs: containerAppUrl responds HTTP 200; record containerAppEnvId and containerAppId

How to Deploy

Prerequisites
  • Azure CLI 2.57+ with Bicep extension
  • Resource Group created in target region
  • Azure Container Registry with image pushed
  • Contributor + User Access Administrator roles
Deployment Steps
  1. Clone repo and navigate to landing-page/specs/
  2. Update image reference in Bicep template
  3. Run az deployment group create (see below)
  4. Capture outputs and assign RBAC roles
  5. Verify FQDN responds and logs appear in Log Analytics
Azure CLI Example
# Deploy Container Apps stack
az deployment group create \
  --resource-group rg-containerapps-prod \
  --template-file azure-containerapps.bicep \
  --parameters environment=prod location=eastus \
  --query "properties.outputs" -o json

FAQ

How is Container Apps different from AKS?

Container Apps abstracts Kubernetes away entirely. AKS gives full Kubernetes control. Use Container Apps for microservices and APIs; use AKS when you need cluster-level customization or advanced scheduling.

Can I enable scale-to-zero?

Yes, set scale.minReplicas: 0 in the Bicep template. Containers spin down when idle and cold-start on the next request. The spec default of minReplicas: 1 avoids cold starts for production APIs.

What image registries are supported?

Azure Container Registry (recommended with managed identity pull), Docker Hub, GitHub Container Registry, and any private registry with username/password credentials stored as secrets.

How do I get logs from my containers?

Logs stream automatically to the Log Analytics workspace linked in the Container App Environment. Query with KQL: ContainerAppConsoleLogs_CL | where ContainerAppName_s == 'app-prod'.

What is Dapr and when should I use it?

Dapr provides portable APIs for state management, pub/sub messaging, service invocation, and secret access. Enable it on your Container App to simplify inter-service communication and avoid tight SDK coupling.
