Microsoft Entra ID

Unified identity, authentication, authorization, workload identity, and governance baseline for Azure platform services and enterprise access control.

View Spec Learn More

Overview

Microsoft Entra ID provides the identity plane for Azure platforms and enterprise applications. It centralizes sign-in, authorization, conditional access, service principals, managed identities, and access governance so teams can apply zero-trust controls consistently across users, workloads, and external collaboration.

Key Capabilities

  • Single Sign-On: Centralized authentication across Microsoft and third-party applications
  • Conditional Access: Policy-based enforcement using user, device, location, and risk signals
  • Managed and Federated Identities: Secretless workload authentication for Azure services, AKS, and CI/CD
  • Role and Group Governance: Group-based access assignment and tenant-wide privilege segmentation
  • Identity Protection: Risk detection for suspicious sign-ins, compromised credentials, and anomalous behavior
  • External Identity Controls: B2B collaboration, guest access lifecycle, and review workflows

Common Use Cases

Privileged Access Baseline

Enforce MFA, compliant device access, and least-privilege controls for administrators and operators.

Platform Workload Identity

Use service principals, managed identities, and OIDC federation for applications and automation.

Enterprise App Onboarding

Standardize app registration, owner assignment, permissions review, and role mapping.

B2B Collaboration Governance

Manage guest users, access packages, and periodic reviews for external identities.

Acceptance Gates
  • Gate 1: App registration, service principal, and security groups are created with approved naming and ownership.
  • Gate 2: Privileged access paths enforce MFA and tenant access policies according to baseline controls.
  • Gate 3: Graph permissions and Azure RBAC assignments are validated for least privilege.

  2. Review: Obtain platform, security, and IAM approval for tenant-scoped changes.
  3. Validate: Confirm group naming, permission sets, and MFA posture before rollout.
  4. Deploy: Execute approved IaC at tenant scope with controlled operator access.
  5. Evidence: Capture sign-in, policy, and assignment validation artifacts for auditability.

How to Deploy

Use this sequence to deploy the Microsoft Entra ID baseline from specification and verify governance and access controls.

1. Prerequisites
  • Azure CLI authenticated with tenant-level rights and Bicep support available
  • Approved application naming, group prefix, and tenant identity governance model
  • Graph permission requirements reviewed by platform and security stakeholders
  • Conditional Access, access review, and logging expectations documented for the target environment
2. Deployment Steps
  1. Review tenant-scope impact and confirm the deployment operator is approved.
  2. Compile and review the Entra ID Bicep template.
  3. Deploy with approved environment, app name, and group prefix inputs.
  4. Verify app registration, enterprise app, and group creation.
  5. Validate downstream RBAC, MFA posture, and workload identity behavior.
3. Azure CLI Example
az bicep build \
  --file specs/azure-entraid.bicep

az deployment tenant create \
  --name entraid-baseline-<environment> \
  --location eastus \
  --template-file specs/azure-entraid.bicep \
  --parameters environment=<environment> appDisplayName=<app-name> groupPrefix=<group-prefix>
4. Validation Checklist
  • Outputs appId, servicePrincipalObjectId, operatorsGroupId, and auditorsGroupId are returned
  • App registration and enterprise application are visible with approved audience and owners
  • Group-based role mapping and least-privilege assignments are confirmed
  • Privileged MFA and workload identity patterns are validated
  • Audit and sign-in monitoring evidence is available to operations teams

FAQ

When should I use managed identity versus an app registration?

Use managed identities for Azure-hosted workloads when possible. Use app registrations when you need cross-platform identity, explicit application audiences, or tenant-level integration patterns.

How should Conditional Access be rolled out?

Start with report-only mode, review sign-in impact, then phase enforcement by role and application criticality to avoid operational lockout.

How do I secure CI/CD access to Azure?

Prefer workload identity federation with branch or subject restrictions instead of storing long-lived client secrets in pipelines.
