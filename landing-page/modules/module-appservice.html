Azure App Service

Managed platform for web apps and APIs with deployment slots, autoscale, VNet integration, managed identity, and production-grade operations.

View Spec Learn More

Overview

Azure App Service provides a fully managed PaaS runtime for web applications and HTTP APIs. It removes infrastructure management overhead while supporting enterprise security controls, CI/CD workflows, deployment slots, autoscale, and deep telemetry integrations.

This specification deploys a Linux App Service Plan, a production web app, a staging slot for blue-green releases, system-assigned managed identity, and Application Insights wiring for centralized observability.

Key Capabilities

  • Managed Runtime: Host APIs and web apps without managing VMs or orchestrators
  • Deployment Slots: Blue-green and canary deployments with near-zero downtime slot swaps
  • Autoscale: Scale up/out based on CPU, memory, and request metrics
  • Managed Identity: Secretless access to Key Vault, Storage, and downstream Azure services
  • Security Baseline: HTTPS-only, FTPS disabled, TLS 1.2 enforced by default
  • Always On: Keeps apps warm for lower latency and stable background execution
  • Observability: Native App Insights telemetry and distributed tracing support
  • Network Controls: VNet integration, private endpoints, and access restrictions

SKU: P1v3 (default)

Acceptance Gates
  • Web app and staging slot deploy successfully with HTTPS-only and TLS 1.2 enforced
  • Managed identity enabled; Key Vault secret retrieval succeeds using RBAC
  • Deployment slot swap from staging to production completes with no downtime
  • Application Insights receives request, exception, and dependency telemetry within 5 minutes
  • Security baseline validated: FTPS disabled, HTTP blocked, and SCM access restricted
Deployment Workflow
  1. Provision Application Insights component for telemetry collection
  2. Deploy Linux App Service Plan with selected SKU and capacity
  3. Deploy production web app with managed identity and security settings
  4. Create staging deployment slot for release validation
  5. Deploy build artifact to staging slot and run smoke tests
  6. Swap staging slot into production and monitor health metrics
  7. Capture outputs and update downstream integrations (APIM, DNS, monitors)

How to Deploy

Prerequisites
  • Azure CLI 2.57+ with Bicep extension
  • Contributor role on target resource group
  • Build artifact/package ready for deployment
  • App Insights and Key Vault governance standards defined
  • Domain/DNS ownership for production hostname
Deployment Steps
  1. Clone repo and navigate to landing-page/specs/
  2. Set plan SKU and runtime stack parameters
  3. Deploy Bicep template and capture outputs
  4. Deploy app build to staging slot
  5. Execute slot swap and monitor post-cutover health
Azure CLI Example
# Deploy App Service baseline
az deployment group create \
  --resource-group rg-appsvc-prod \
  --template-file azure-appservice.bicep \
  --parameters environment=prod appServicePlanSkuName=P1v3 appRuntimeStack=PYTHON|3.11 \
  --query "properties.outputs" -o json

# Swap staging slot to production
az webapp deployment slot swap \
  --resource-group rg-appsvc-prod \
  --name appsvc-prod \
  --slot staging \
  --target-slot production

FAQ

When should I choose App Service over AKS or Container Apps?

Choose App Service for mainstream web/API workloads where you want fast PaaS deployment, deployment slots, and minimal platform operations. Choose AKS for advanced orchestration needs and Container Apps for event-driven container workloads with scale-to-zero.

Why use deployment slots in production?

Slots let you deploy and validate new versions in an isolated staging environment using production-like settings, then swap with almost zero downtime. Rollback is immediate by swapping back.

How do I handle secrets safely?

Enable managed identity on the app and reference Key Vault secrets in app settings using Key Vault references or SDK-based retrieval. Avoid hardcoding secrets in code or pipeline variables.

Can App Service be private-only?

Yes. Use private endpoints for inbound access, VNet integration for outbound private connectivity, and access restrictions to block public traffic paths. Pair with WAF/APIM for controlled ingress.

What are the top cost levers?

Primary levers are plan SKU, instance count, autoscale thresholds, and telemetry ingestion volume. Right-size instance class, scale schedules, and logging verbosity per environment.
