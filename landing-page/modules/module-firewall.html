Azure Firewall

Managed, stateful network firewall for enterprise-grade ingress and egress control with threat intelligence and centralized policy governance.

🛡️

Overview

Azure Firewall is a cloud-native, highly available network security service for filtering east-west and north-south traffic. It supports L3-L7 policies, application FQDN rules, DNAT/SNAT, and threat-intelligence based controls with centralized monitoring.

Key Capabilities

  • Stateful L3-L7 Inspection: Evaluate traffic flows using network, application, and NAT rule collections
  • Threat Intelligence: Alert or deny traffic to known malicious IPs and domains
  • Policy Hierarchy: Centralized management with Firewall Policy and reusable rule collection groups
  • TLS Inspection (Premium): Decrypt and inspect outbound HTTPS flows for advanced filtering
  • IDPS (Premium): Signature-driven intrusion detection and prevention
  • Zone Redundancy: Multi-zone deployment support for resilience and continuity

Common Use Cases

🏗️ Hub-and-Spoke Security

Centralize network controls in a hub VNet and inspect transit traffic from spoke workloads.

🌐 Controlled Internet Egress

Route outbound traffic through approved FQDN and destination rules with full audit trails.

🧠 Threat Intelligence Filtering

Block communication with known malicious endpoints to reduce lateral and external risk.

🔒 Zero Trust Enforcement

Apply explicit allow-list rules and deny-by-default posture for sensitive workloads.

How to Deploy

Use this workflow to deploy Azure Firewall from the parameter-driven specification and validate the environment before traffic cutover.

1. Prerequisites
  • Azure CLI with Bicep support installed and authenticated with the target subscription
  • An existing virtual network with a dedicated AzureFirewallSubnet
  • Approved values for location, environment, firewallSkuTier, threatIntelMode, and baseline rule collections
  • Log Analytics workspace ready for firewall diagnostics
2. Deployment Steps
  1. Create or select the resource group: Ensure the target resource group matches the approved environment boundary.
  2. Confirm network dependencies: Verify the VNet, AzureFirewallSubnet, route tables, and public IP requirements are already approved.
  3. Build the Bicep template: Compile the spec artifact before running the deployment to catch validation issues early.
  4. Deploy with parameter values: Pass the approved location and environment values and keep rule collection values aligned with the spec.
  5. Validate post-deploy: Confirm firewall health, logging, routing, and rule enforcement before moving production traffic.
3. Azure CLI Example
az account set --subscription <subscription-id>

az group create \
  --name rg-network-<environment> \
  --location <location>

az bicep build \
  --file specs/azure-firewall.bicep

az deployment group create \
  --resource-group rg-network-<environment> \
  --template-file specs/azure-firewall.bicep \
  --parameters location=<location> environment=<environment>
4. Validation Checklist
  • Public IP and private IP are provisioned and visible in deployment outputs
  • Threat intelligence mode is set to the approved state
  • Firewall logs are flowing to Log Analytics
  • Rule collections enforce expected allow/deny behavior
  • User-defined routes send intended traffic through the firewall path

FAQ

Can I use Azure Firewall with NSGs?

Yes. NSGs and Azure Firewall are complementary. NSGs secure subnet/NIC boundaries while Firewall provides centralized policy and advanced traffic inspection.

When should I choose Premium tier?

Choose Premium when you require TLS inspection, IDPS, and advanced URL filtering for stricter security and compliance workloads.

How do I validate forced tunneling?

Validate UDRs and next-hop paths from workload subnets, then confirm all inspected flows are visible in firewall diagnostics.
