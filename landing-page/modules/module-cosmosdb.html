Azure Cosmos DB

Globally distributed, low-latency database platform for mission-critical workloads requiring elastic scale, high availability, and flexible data models.

View Spec Learn More

Overview

Azure Cosmos DB is a fully managed, globally distributed database service built for internet-scale applications. It provides predictable performance, enterprise availability, and multi-region resiliency while supporting diverse workload patterns across document, key-value, graph, and columnar models.

Key Capabilities

  • Global Distribution: Deploy data close to users with regional replication and automatic failover options
  • Multi-Model APIs: SQL (Core), MongoDB, Cassandra, Gremlin, and Table interfaces
  • Guaranteed Performance: Consistent low-latency reads and writes with throughput controls
  • Consistency Choices: Strong, bounded staleness, session, consistent prefix, and eventual
  • Elastic Scale: Scale throughput and storage as workloads evolve
  • Continuous Backup: Built-in backup and restore capabilities for operational resilience

Common Use Cases

Global SaaS Data Layer

Serve distributed users with low-latency access and high availability across regions.

High-Volume User Profiles

Store identity, preferences, and session state for large-scale consumer platforms.

Operational Event Stores

Persist event-centric application records with flexible schema and predictable throughput.

Real-Time Product Catalogs

Power low-latency reads for dynamic catalogs, personalization, and recommendation workflows.

How to Deploy

Use this sequence to deploy Azure Cosmos DB from specification and verify performance and resilience controls.

1. Prerequisites
  • Azure CLI with Bicep support installed and authenticated to target subscription
  • Approved location, environment, partition key, and throughput values
  • Network and access policy decisions finalized (public network access, identity, RBAC)
  • Operational monitoring and backup policy aligned with governance requirements
2. Deployment Steps
  1. Create or select the data services resource group.
  2. Compile and review the Cosmos DB Bicep template.
  3. Deploy with approved parameter set for environment and location.
  4. Run connectivity and throughput baseline tests.
  5. Validate diagnostics, backup settings, and role-based access controls.
3. Azure CLI Example
az account set --subscription <subscription-id>

az group create \
  --name rg-cosmos-<environment> \
  --location <location>

az bicep build \
  --file specs/azure-cosmosdb.bicep

az deployment group create \
  --resource-group rg-cosmos-<environment> \
  --template-file specs/azure-cosmosdb.bicep \
  --parameters location=<location> environment=<environment>
4. Validation Checklist
  • Outputs cosmosDbId, cosmosDbEndpoint, cosmosDbName, and databaseName are returned
  • Client connectivity and authentication flows are verified
  • Throughput and partition design meet expected workload profile
  • Backup and restore posture is tested against policy
  • Diagnostic telemetry and alerts are visible for operations teams

FAQ

How should I choose a partition key?

Choose a key with high cardinality and balanced access distribution to avoid hot partitions and sustain throughput efficiency.

When should autoscale be used?

Autoscale is a strong choice for workloads with significant traffic variation where manual RU/s tuning is operationally expensive.

What consistency level is recommended by default?

Session consistency is often a practical baseline for balancing performance and correctness, but final selection should align with workload semantics.
