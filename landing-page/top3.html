Centralized Hub AI

Pattern 1 — Shared AI platform in the Hub, consumed by multiple spokes

Full Details

All AI services — Azure OpenAI, AI Search, Machine Learning, and supporting data/security services — deployed in a single Hub VNet. Application spokes connect via VNet Peering to consume shared services through Private Endpoints.

How This Pattern Works
Architecture Concept

The Centralized Hub AI pattern follows the classic enterprise hub-and-spoke network topology, placing all AI services in a single, well-governed Hub VNet. Think of the Hub as your organization’s “AI control tower” — it hosts Azure OpenAI (GPT-4o at 30K tokens-per-minute plus embedding models at 120K TPM), Azure AI Search with vector and semantic indexes, Azure Machine Learning for model management, and all supporting infrastructure (Key Vault, ADLS Gen2, Log Analytics).

Application teams deploy their workloads (AKS clusters, App Services, Container Apps) in spoke VNets that peer back to the Hub. When a spoke application needs to call GPT-4o or query AI Search, traffic flows over VNet Peering through Private Endpoints — never touching the public internet. Every API call is private, auditable, and secured by NSG rules.

Typical Data Flow
  1. User request → Spoke App Service
  2. VNet Peering → Hub Private Endpoint
  3. Azure OpenAI (with RAG retrieval from AI Search)
  4. Response returns via same private path
  5. All activity logged to central Log Analytics workspace
Why Centralize?

Azure OpenAI deployments are expensive (GPU-backed, TPM-metered). When each team provisions its own instance, you pay N times for underutilized quota. A centralized deployment lets you pool quota across all consumers, achieving 40–60% cost savings while maintaining a single governance layer — one place to manage model versions, prompt templates, content filters, and usage policies.

Trade-offs
  • Works best when teams share similar AI needs and don’t require isolated data stores
  • If departments need separate vector databases with strict data boundaries (FERPA, HIPAA), consider the Hybrid pattern instead
  • Hub is a single point of management (advantage) but also a shared dependency
  • Teams share TPM quota and must coordinate on model upgrades

Hybrid Central-Spoke AI

Pattern 3 — Shared LLM in Hub + per-spoke RAG components

Full Details

Core AI services centralized in the Hub (base models, model registry) while inference, customization, and RAG components are deployed in application spokes. Balances cost efficiency with workload autonomy.

How This Pattern Works
Architecture Concept

The Hybrid Central-Spoke pattern splits the AI stack across two layers:

Hub (Shared, Expensive)

  • Azure OpenAI — GPT-4o at 50K TPM, embeddings at 120K TPM
  • AML Model Registry — versioning and staged rollouts
  • Central Key Vault — shared API keys
  • Log Analytics — cross-spoke observability

Spoke (Local, Data-Adjacent)

  • Azure AI Search — local vector DB for RAG
  • ADLS Gen2 — source documents per team
  • App Service / AKS / Container Apps — compute

When a spoke app receives a query, it retrieves context from its local AI Search index, then sends the enriched prompt to the shared GPT-4o in the Hub via VNet Peering. The LLM never sees raw documents from other spokes.

Why Hybrid?

Many organizations need the cost efficiency of shared LLM quota but can’t co-mingle data across departments, campuses, or business units. FERPA, HIPAA, PCI-DSS, and data residency laws often mandate that documents and indexes stay within a team’s own network boundary. The Hybrid pattern solves this by keeping data local while sharing the most expensive resource (the LLM).

Trade-offs
  • Per-spoke AI Search + storage costs ~$250/mo (relatively cheap)
  • Save 60%+ on LLM costs by sharing one OpenAI deployment
  • Adding a new spoke is incremental — deploy, peer to Hub, done in 1–2 weeks
  • Hub team manages model versions centrally via AML Model Registry
Typical Data Flow
  1. User query → Spoke App Service
  2. Spoke AI Search (vector retrieval from local docs)
  3. Enriched prompt → VNet Peering → Hub Azure OpenAI
  4. GPT-4o inference → response returns to spoke
  5. Both spoke and hub logs → central Log Analytics

Greenfield Azure Deployment

Pattern 10 — Zero to Hero with Cloud Adoption Framework

Full Details

Build a brand-new Azure environment from zero using Cloud Adoption Framework best practices. Complete landing zone setup with management groups, networking, identity, governance, and AI-ready infrastructure.

How This Pattern Works

The Greenfield Azure Deployment pattern is for organizations starting from zero on Azure — no existing subscriptions, no VNets, no identity configuration. Instead of organically building infrastructure and accumulating technical debt, this pattern deploys the full Cloud Adoption Framework (CAF) Enterprise-Scale Landing Zone in one coordinated effort, then layers AI services on top.

6 CAF Phases
  1. Strategy — Define AI workloads, business outcomes, and compliance requirements (FERPA, SOC 2, HIPAA)
  2. Plan — Design subscription topology and management group hierarchy
  3. Ready — Deploy Platform subscriptions (Management, Connectivity, Identity) and Landing Zone subscriptions
  4. Adopt — Build hub-spoke networking with Azure Firewall, Bastion, VPN Gateway, and Private DNS Zones
  5. Govern — Apply Azure Policy initiatives, enable Defender for Cloud, and configure cost management budgets
  6. Manage — Set up Log Analytics, monitoring alerts, and operational runbooks
Why Greenfield?

Organizations in regulated industries (healthcare, education, government, financial services) need audit-ready governance from day one. Retro-fitting management groups, policies, and network security onto an organically grown Azure environment typically costs 6–12 months of rework and significant re-architecture. The greenfield approach eliminates this by building the foundation correctly from the start.

What’s Included
  • Management group hierarchy (Tenant Root → Organization → Platform + Landing Zones)
  • Hub VNet with Azure Firewall (egress filtering + TLS inspection)
  • Azure Bastion for secure admin access
  • Entra ID with Conditional Access policies
  • Azure Policy initiatives (deny public endpoints, enforce tagging, require encryption)
  • Defender for Cloud (security posture management)
  • Cost Management with pre-configured budgets and alerts
AI-Ready from Day One

The AI spoke is pre-provisioned with Azure OpenAI, AI Search, Key Vault, ADLS Gen2, and App Service — all connected via Private Endpoints through the Hub firewall. New AI workloads deploy into the existing landing zone structure without any infrastructure changes. The environment scales from 1 to 100+ subscriptions using the same management group and policy framework.

Executive Summary
Reduce AI Infrastructure Costs

Shared AI platforms eliminate per-team duplication of expensive services like Azure OpenAI and AI Search. Organizations typically see 40-60% cost savings by consolidating LLM deployments while maintaining workload isolation.

Accelerate Time-to-Value

Pre-built landing zones with Bicep IaC templates reduce deployment from months to days. New teams onboard to existing AI platforms without provisioning infrastructure, enabling 3-6x faster delivery of AI use cases.

Mitigate Security & Compliance Risk

Private endpoints, Azure Firewall, NSGs, and Defender for Cloud are embedded in every pattern. Zero public endpoints for AI services, full audit trails, and policy-as-code governance satisfy SOC 2, HIPAA, and PCI-DSS requirements.

Optimize Team Productivity

Central platform teams manage shared infrastructure (2-3 FTEs) while application teams focus on building AI features. Eliminates redundant ops work across business units and reduces mean-time-to-resolution for AI service issues.

Enable Scalable Growth

All three patterns are designed to scale from 1 to 100+ workloads without re-architecture. Management group hierarchies, hub-spoke networking, and IaC templates ensure consistent, repeatable deployments as the organization grows.

CSI — Centralized IQ Engine Platform

A shared GPT-4o instance (30K TPM) serves the AI Academic Advisor, Dropout Risk Predictor, and Personalized Learning Planner through Private Endpoints.

Case Study 2

CSI — Multi-Campus University System

Industry: Higher Education  |  Campuses: 8 institutions, 120K students  |  Pattern: Hybrid Central-Spoke

Challenge

A state university system with 8 campuses needed CSI's AI platform deployed across all institutions. Each campus had its own LMS (mix of Canvas, Blackboard, and Moodle), its own SIS, and different academic catalogs. FERPA regulations prohibit sharing student records across campuses. A single centralized deployment would violate data governance policies; deploying 8 complete platforms would cost 4x the budget.

Solution

Deployed Hybrid Central-Spoke with shared GPT-4o (50K TPM) and the AML Model Registry in the Hub. Each campus spoke runs its own Work IQ crawler (connected to its local LMS), its own Fabric IQ pipeline (connected to its SIS), and its own AI Search indexes. Foundry IQ runs in both Hub (shared accreditation standards, state policies) and spokes (campus-specific catalogs, local policies). Student data never crosses campus VNet boundaries.
