1Core Concept

Specifications are primary, executable artifacts — not passive documentation. They act as enforcement mechanisms in the development lifecycle. If implementation deviates from specs, builds fail automatically.
SDD prevents:
Architectural Drift

Service boundaries erode over time without enforcement

API Contract Violations

Breaking changes slip through without contract checks

Security Inconsistencies

Security patterns applied unevenly across services

2Why It Matters — Especially with AI Development

Problem: AI-Generated Code
  • Often passes unit tests
  • Can violate architecture constraints
  • May break API contracts silently
  • May introduce security gaps
Solution: SDD Introduces
  • Validation gates at build time
  • System-level governance for AI code
  • Cross-service contract enforcement
  • Automated compliance checks
Traditional testing validates functionality. SDD enforces system-wide constraints. Both are necessary — they solve different problems.

3Key Problem It Solves

Specification Drift

Specs and implementation diverge across services and teams over time

No Cross-Service Enforcement

Architecture rules exist in docs but aren't enforced in CI/CD

Hidden Integration Issues

API mismatches, security gaps, and distributed system failures surface late

4Foundational Requirements

Architecture
  • Clearly defined service boundaries
  • Standardized API contracts (REST/GraphQL schemas)
  • Multi-service or distributed system awareness
Platform
  • CI/CD pipeline capable of running validation checks
  • Build failure on spec violations
  • Multi-environment promotion gates
Tooling
  • Specification frameworks or formats
  • AI-assisted development tools (optional)
  • Contract testing libraries

7Standard Workflow — 5 Phases

1
Define Specifications
  • Capture requirements, constraints, success criteria
  • Must be machine-readable and executable
  • API schemas, architecture rules, security policies
2
Plan Implementation
  • Translate specs into architecture decisions
  • Select technology choices and design patterns
  • Map specs to service boundaries
3
Break into Tasks
  • Decompose into isolated, testable units
  • Each task linked to a spec section
  • Clear acceptance criteria from spec
4
Implement with Constraints
  • Code (human or AI) must adhere to specs
  • Validation gates run at build time
  • Violations = build failure
5
Debug the Spec, Not Just Code
  • Root cause of failures = spec gaps
  • Fix specs → propagates across all implementations
  • Specs evolve as the system evolves

10Limitations

SDD is not ideal for:
  • Exploratory / R&D work — specs constrain experimentation
  • Rapid prototyping — high upfront overhead
  • Very small teams — high relative cost vs. benefit
  • Poorly documented legacy systems — hard to extract specs

12Strategic Takeaway

Before

"Code first, test later"

Specs as passive documentation that nobody reads

After

"Spec first, enforce continuously"

Specs as active system controls that break the build

Treat specifications as active system controls, not passive documentation.

Every constraint in the spec is a test that runs on every build. Every gap in the spec is a vulnerability waiting to surface in production.

13The Power of .md as a Spec Format

Markdown is the most underestimated specification format. It's human-readable, version-controllable, diff-friendly, and parseable by AI agents — making it the ideal bridge between business intent and machine execution.
Why .md Works for Specs
  • Git-native — full version history, branching, PR reviews on specs
  • Diff-friendly — line-by-line change tracking (unlike JSON/YAML blobs)
  • Human-readable — execs, architects, and developers all understand it
  • AI-parseable — LLMs natively understand Markdown structure
  • Zero tooling — no IDE plugins or compilers needed
  • Embeds code — fenced blocks for Bicep, JSON, YAML, HCL inline

14Azure Infrastructure — Spec Samples

Real-world examples of Azure components defined as executable .md specifications. Each spec is self-contained — an AI agent or engineer can build the entire component from the spec alone.

Sample 1 — APIM GenAI Gateway 
**Document ID:**   DS-APIM-001
**Version:**       1.0
**Status:**        Approved
**Type:**          Deployment
**Owner:**         <platform-team>
**Date:**          2026-03-19
**Applies to:**    API Management GenAI Gateway

---

## 1. Overview
API Management v2 (Standard) acting as a GenAI gateway for Azure OpenAI.
Routes, rate-limits, caches, and circuit-breaks all LLM traffic.

## 2. Scope
APIM instance, GenAI policies, backend pool, semantic caching, and circuit-breaker.
Out of scope: Azure OpenAI provisioning (see DS-AOAI-001), VNet hub (see DS-NET-001).

## 3. Requirements
**FR-01** — APIM must use Standard v2 SKU (supports GenAI policies).
**FR-02** — APIM must enforce token rate limit of 80K TPM per subscription key.
**FR-03** — APIM must enable semantic caching (Azure AI Search backend).
**FR-04** — APIM must implement circuit breaker: 5 failures in 60s → open for 30s.
**NFR-01** — Region: East US 2.
**NFR-02** — P99 latency for cache-hit requests must be <50ms.

## 4. Architecture
- Backend pool: 2× Azure OpenAI endpoints (East US 2 + West US 3)
- Load balancing: round-robin with priority failover
- Private Endpoint in hub VNet, PE subnet /27
- Data flow: Client → APIM (PE) → semantic cache check → AOAI backend

## 5. Security
- Authentication: Subscription key required for all consumers
- Authorization: Managed Identity → Azure OpenAI (Cognitive Services OpenAI User)
- Secrets: Managed Identity → Key Vault (Key Vault Secrets User) — no stored secrets
- Encryption in transit: TLS 1.2 minimum, mTLS for backend connections
- Encryption at rest: Platform-managed keys (PMK)
- Data classification: Confidential (LLM prompts and completions)
- Compliance: SOC 2, ISO 27001

## 6. Network
- Private Endpoint in hub VNet PE subnet (/27)
- No public endpoint — private access only
- DNS: privatelink.azure-api.net zone linked to hub VNet

## 7. Cost

|-------------------|-----------------|-------------|------------|--------------|
| APIM Standard v2  | 1 unit          | $700/unit   | 1          | $700.00      |
| Total             |                 |             |            | $700.00      |
| + 20% buffer      |                 |             |            | $840.00      |

Budget ceiling: $840/mo. Alert at 80% ($672) and 100% ($840).

- Configuration: APIM config export → re-import previous revision
- Data: N/A (stateless gateway)

- [ ] AC-01 — APIM responds on private endpoint only (no public access)
- [ ] AC-02 — Token rate limit enforced (test with 100K TPM burst → HTTP 429)
- [ ] AC-03 — Semantic cache hit ratio > 0% after warm-up
- [ ] AC-04 — Circuit breaker triggers on backend failure simulation
- [ ] AC-05 — Managed Identity has no Key Vault secret list permission

## 1. Overview
Central AI Foundry workspace (Hub) with one Project for the ML team.
Includes Prompt flow, AI Evaluation, and model catalog access.

## 2. Scope
Hub workspace, Project workspace, connected services, and compute.
Out of scope: Azure OpenAI provisioning (see DS-AOAI-001), network hub (see DS-NET-001).

## 3. Requirements
**FR-01** — Hub workspace must use managed VNet (AllowInternetOutbound).
**FR-02** — Project workspace must inherit Hub networking.
**FR-03** — Hub must connect to: Azure OpenAI, AI Search, Storage, Key Vault.

- Data classification: Confidential (training data and model weights)

## 6. Network

Sample 3 — Hub-Spoke Network + Firewall 
**Document ID:**   DS-NET-001
**Version:**       1.0
**Status:**        Approved
**Type:**          Deployment
**Owner:**         <network-team>
**Date:**          2026-03-19
**Applies to:**    Hub-Spoke Network with Azure Firewall

---

## 1. Overview
Hub VNet with Azure Firewall, Bastion, and VPN Gateway.
All spoke traffic routes through the hub via UDR 0.0.0.0/0.

## 2. Scope
Hub VNet, subnets, Azure Firewall, Bastion, VPN Gateway, NSGs, UDRs, and DNS.
Out of scope: spoke VNet provisioning (see DS-SPOKE-001), AI workloads.

## 3. Requirements
**FR-01** — Hub VNet must use address space 10.0.0.0/16.
**FR-02** — Azure Firewall must be Standard SKU with DNS proxy enabled.
**FR-03** — Bastion must be Standard SKU (file transfer, native client).
**FR-04** — VPN Gateway must be VpnGw1 (P2S for admin access).
**NFR-01** — All spoke-to-internet traffic must traverse Azure Firewall.

## 4. Network
Network option: Private (VNet + Private Endpoints)

| Subnet Name            | CIDR           | Purpose               | Min Size | NSG           |
|------------------------|----------------|-----------------------|----------|---------------|
| AzureFirewallSubnet    | 10.0.1.0/26    | Azure Firewall        | /26      | N/A (managed) |
| AzureBastionSubnet     | 10.0.2.0/26    | Azure Bastion         | /26      | N/A (managed) |
| GatewaySubnet          | 10.0.3.0/27    | VPN Gateway           | /27      | N/A (managed) |
| PrivateEndpointSubnet  | 10.0.4.0/24    | Private Endpoints     | /24      | nsg-pe        |
| ManagementSubnet       | 10.0.5.0/24    | Jump boxes / mgmt     | /24      | nsg-mgmt      |

CIDRs are non-overlapping, sized with 30%+ headroom.

## 5. Architecture
- Spoke VNets peer to Hub (allow forwarded traffic)
- UDR on every spoke subnet: 0.0.0.0/0 → Firewall private IP
- Firewall rules: allow HTTPS, deny all else by default
- Firewall application rules: allow *.openai.azure.com, *.cognitiveservices.azure.com
- DNS: Azure Firewall as DNS proxy → Azure DNS

## 6. Security
- Authentication: Entra ID for VPN P2S; Bastion for VM access
- Authorization: Network Contributor for hub ops; Reader for spoke teams
- Secrets: VPN root cert in Key Vault; no shared passwords
- Encryption in transit: TLS 1.2+ on all management plane; IPsec for VPN
- Encryption at rest: NSG flow logs encrypted in Storage (PMK)
- Data classification: Internal (network metadata)
- Compliance: SOC 2, ISO 27001, CIS Azure Benchmark

NSG rules (nsg-pe example):
| Priority | Name         | Direction | Protocol | Source      | Destination | Port  | Action |
|----------|------------- |-----------|----------|-------------|-------------|-------|--------|
| 100      | AllowVnet    | Inbound   | *        | VirtualNet  | VirtualNet  | *     | Allow  |
| 4096     | DenyAllIn    | Inbound   | *        | *           | *           | *     | Deny   |

Sample 5 — Confidential AI (Zero Trust) 
**Document ID:**   SEC-CAI-001
**Version:**       1.0
**Status:**        Approved
**Type:**          Architecture
**Owner:**         <security-team>
**Date:**          2026-03-19
**Applies to:**    Confidential AI (Zero Trust) Environment

---

## 1. Overview
Confidential computing environment for regulated AI workloads.
All data encrypted in use (memory), at rest, and in transit.

## 2. Scope
Confidential VMs, Managed HSM, Firewall Premium, DDoS Protection, Confidential Ledger.
Out of scope: AI model training logic, application code, data ingestion pipelines.

## 3. Requirements
**FR-01** — All compute must use DCesv5 (AMD SEV-SNP) for confidential processing.
**FR-02** — Key management must use Managed HSM (FIPS 140-2 Level 3).
**FR-03** — Azure Firewall must be Premium SKU (IDPS + TLS inspection).
**FR-04** — Confidential Ledger must record all sensitive operations (immutable).
**NFR-01** — No service may have a public endpoint.
**NFR-02** — TLS 1.3 must be enforced end-to-end.

## 4. Network
Network option: Private (Fully isolated VNet)

| Subnet Name            | CIDR           | Purpose               | Min Size | NSG               |
|------------------------|----------------|-----------------------|----------|--------------------|
| AzureFirewallSubnet    | 10.2.0.0/26    | Firewall Premium      | /26      | N/A (managed)      |
| snet-confidential      | 10.2.1.0/24    | DCesv5 VMs            | /24      | nsg-deny-all       |
| snet-pe                | 10.2.2.0/24    | Private Endpoints     | /24      | nsg-pe             |
| AzureBastionSubnet     | 10.2.3.0/26    | Bastion               | /26      | N/A (managed)      |

8+ Private DNS zones for PE resolution. DNS: Private DNS Zones linked to VNet.

## 5. Architecture
- Isolated VNet: all PaaS behind Private Endpoints
- Azure Firewall Premium at the perimeter (IDPS + TLS inspection)
- AI Foundry Hub: managed VNet isolation mode
- Confidential Ledger for tamper-proof operation logs
- Data flow: User → Bastion → CVM → PE → AOAI (all private, all encrypted)

## 6. Security
- Authentication: Entra Workload ID — secretless authentication everywhere
- Authorization: Least-privilege RBAC per component; HSM Crypto Officer for key ops
- Secrets: No secrets in Key Vault for service auth — Workload ID only; HSM for keys
- Encryption in transit: TLS 1.3 enforced end-to-end
- Encryption at rest: CMK mandatory via Managed HSM; Storage: CMK + infrastructure double encryption
- Encryption in use: AMD SEV-SNP memory encryption on DCesv5
- Data classification: Restricted (regulated AI workloads, PII, PHI)
- Compliance: SOC 2, ISO 27001, HIPAA, FedRAMP, PCI-DSS, NIST 800-53

Zero Trust checklist:
- [ ] All service-to-service calls use Managed Identity — no shared secrets
- [ ] Least-privilege RBAC roles documented per component
- [ ] Network access restricted to required flows only (DenyAllInbound)
- [ ] All data classified and handling rules documented
- [ ] Audit logging enabled on all sensitive operations (Confidential Ledger)
- [ ] MFA enforced for human access to production systems

| Bastion              | Standard           | $329/mo     | 1          | $329.00      |
| Other (Storage, PE)  | Various            | —           | —          | $389.00      |
| Total                |                    |             |            | $10,038.00   |
| + 20% buffer         |                    |             |            | $12,045.60   |

Budget ceiling: $13,000/mo. Alert at 80% ($10,400) and 100% ($13,000).
Top cost drivers: HSM ($3,288, 33%), DDoS ($2,944, 29%), Firewall ($1,460, 15%).
Security spend is non-negotiable — do not reduce.

## 8. Rollback
- Infrastructure: re-deploy previous Git tag; mirror VNet + PE + DNS in DR site
- Application: model rollback via ACR image tag
- Data: HSM backup to secondary region with secure transport; Ledger is append-only
- Configuration: DDoS Plan is global; Firewall Policy is portable

## 9. Acceptance Criteria
- [ ] AC-01 — No service has a public endpoint (scan all resources)
- [ ] AC-02 — HSM key rotation succeeds without downtime
- [ ] AC-03 — DCesv5 attestation report validates SEV-SNP
- [ ] AC-04 — Confidential Ledger entry is immutable (tamper test fails)
- [ ] AC-05 — Firewall IDPS blocks known malicious payload
- [ ] AC-06 — Entra Workload ID — no secrets in Key Vault for service auth

## 10. Glossary
- CVM: Confidential Virtual Machine
- HSM: Hardware Security Module
- SEV-SNP: AMD Secure Encrypted Virtualization — Secure Nested Paging
- IDPS: Intrusion Detection and Prevention System
- CMK: Customer-Managed Key
- PHI: Protected Health Information
One .md file per component. The entire Azure environment defined in a folder of specs.

AI agents read the spec → generate Bicep/Terraform → CI validates against the spec → only compliant code deploys. The spec is the contract. The spec is the test. The spec is the source of truth.

15Universal Spec Ruleset

The single source of truth for how specs are written, structured, reviewed, and maintained. Every specification — functional, deployment, architecture, data, or product — must comply before engineering handoff.

SPEC-RULES-001 · v1.0 · Active
Applies to all specification documents. Uses RFC 2119 keywords: must, must not, should, should not, may.

1.1 Required Header

Every spec must begin with this header block — no field may be left blank:

**Document ID:**   <PREFIX>-<DOMAIN>-<NNN>
**Version:**       <MAJOR.MINOR>
**Status:**        Draft | Review | Approved | Deprecated
**Type:**          Functional | Deployment | Architecture | Data | Product
**Owner:**         <name or team>
**Date:**          <YYYY-MM-DD>
**Applies to:**    <system, service, or feature name>
1.2 Document ID Format
PrefixUse
SPECGeneral / product specifications
FSFunctional specifications
DSDeployment specifications
ASArchitecture specifications
SECSecurity specifications

Full format: <PREFIX>-<DOMAIN>-<NNN> — e.g. FS-AVM-001. Addenda: <PARENT-ID>-ADD-<NNN>

1.3 Version Numbering
Change TypeVersion Bump
Structural section added/removedMajor (x.0)
Content updated, rule changedMinor (x.y)
Typo, formatting fixNo bump — update date only
1.4 Status Definitions
StatusMeaningWho Can Set
DraftBeing written — not ready for engineeringAuthor
ReviewComplete — pending stakeholder sign-offAuthor
ApprovedSigned off — safe to build againstOwner + reviewer
DeprecatedSuperseded — do not build againstOwner

A spec cannot advance to Approved without at least one reviewer who is not the author.

2.1 Required Sections

Every spec must contain all of the following. Sections may be marked N/A with a reason but must not be silently omitted.

#SectionRequired In
1Purpose / OverviewAll specs
2ScopeAll specs
3StakeholdersAll specs
4Functional / Technical RequirementsFunctional, Deployment, Architecture
5Data SchemasFunctional, Data specs
6Error HandlingFunctional, Deployment specs

2.2 Table of Contents

Every spec with more than 5 sections must include a linked Table of Contents immediately after the header block.

2.3 Section Numbering
  • Top-level: ## 1. Title
  • Subsections: ### 1.1 Title
  • Maximum three levels deep
2.4 One Document, One Responsibility

Each spec covers one system, service, or feature. Multi-system specs must be split. Addenda (-ADD-NNN) augment — they do not replace parent sections.

3.1 Language
RuleRationale
Use must, must not, should, should not, may (RFC 2119)Removes ambiguity about mandatory vs. optional
Write in active voice"The system must send an alert" not "An alert must be sent"
Use present tense for requirements"The API returns…" not "The API will return…"
No jargon without definitionEvery acronym must appear in the Glossary
No vague quantifiersNever write "fast", "soon", "large" without a measurable value
3.2 Prohibited Phrases
BannedReplace With
"As soon as possible"Specific SLA or time target
"Best effort"Quantified availability or throughput target
"TBD" / "TBC"Placeholder with owner + due date
"etc." / "And so on"Complete the list
"Fast" / "efficient"Specific latency, throughput, or time target
"Scalable"Specific scale target (e.g. 10,000 concurrent users)
"Secure"Specific security control (e.g. TLS 1.2 minimum)
"Simple" / "Obviously"Omit — not measurable
3.3 Placeholder Policy
<placeholder-name>                    e.g. <subscription-id>
<placeholder-name:type>               e.g. <budget-limit:USD>
<placeholder-name:owner:due-date>     e.g. <api-key:@devops:2026-04-01>

A spec with more than 5 unresolved placeholders must not advance to Approved.

3.4 Requirements Format
<ID> — <Component> must <requirement>.
Rationale: <why this requirement exists>

Example:
FR-04 — The API must return a response within 200ms at P95 under normal load.
Rationale: User-facing requests exceeding 200ms measurably increase abandonment rate.

4.1 No Ambiguous Architecture

Every architecture diagram must be accompanied by a written description showing: all services/components, direction of data flow, where authentication occurs, and where data is persisted.

4.2 API Specifications

Any API referenced in a spec must include:

  • Full endpoint URL (or URL template with placeholders)
  • HTTP method + authentication method
  • Required parameters + response schema (key fields and types)
  • Rate limits (if applicable) + link to official documentation
4.3 No Assumed Defaults

Every configurable value must be explicitly stated — defaults change between versions. Specify: SKU/tier, region, replication factor, retention period, timeout values, retry counts and backoff strategy.

4.4 Technology Choices Must Be Justified
| Technology | Choice     | Rationale                                          |
|------------|------------|-----------------------------------------------------|
| Database   | PostgreSQL | ACID compliance required; team has existing expertise |

5.1 All Schemas Must Be Typed

Every schema must include: field name, data type, required/optional, constraints, and an example value.

5.2 Schema Format
{
  "restaurant_id": "UUID        // required — v4 UUID",
  "rating":        "float       // optional — range: 0.0–5.0 — example: 4.2",
  "price_range":   "enum        // required — values: $ | $$ | $$$",
  "last_seen":     "datetime    // required — ISO 8601 UTC"
}
5.3 No Unnamed Fields

Schemas must not contain unnamed fields, wildcard keys, or object types without definition.

5.4 Enum Values Must Be Exhaustive

Every enum must list all possible values. Note if extensible: // extensible enum — new values require schema version bump.

6.1 Mandatory Security Section
AreaRequired Content
AuthenticationMechanism (Entra ID, OAuth, JWT, API key) + token lifetime
AuthorizationRBAC roles, permission model, least-privilege rationale
Secrets ManagementWhere secrets live + rotation policy
Encryption in TransitTLS version minimum, protocols enforced
Encryption at RestMethod (PMK/CMK), key management
Data ClassificationSensitivity level per data type
ComplianceFrameworks applicable (SOC 2, ISO 27001, GDPR, HIPAA)
6.2 No Hardcoded Credentials

Specs must never contain real credentials, connection strings, API keys, or tokens — even as examples.

6.3 No "Security TBD"

Security requirements must be fully specified before Approved status. Security is not a post-deployment concern.

6.4 Zero Trust Checklist
[ ] All service-to-service calls use Managed Identity — no shared secrets
[ ] Least-privilege RBAC roles documented per component
[ ] Network access restricted to required flows only
[ ] All data classified and handling rules documented
[ ] Audit logging enabled on all sensitive operations
[ ] MFA enforced for human access to production systems

7.1 No Unitemised Cost Estimates

A total monthly cost must never appear without a line-item breakdown: service name, SKU/config, unit price, estimated usage, monthly cost, source.

7.2 Cost Estimate Format
| Service   | SKU / Config | Unit Price | Est. Usage | Monthly Cost |
|-----------|-------------|-----------|-----------|-------------|
| <name>    | <sku>       | $x.xx/unit | <volume>  | $xx.xx      |
| Total     |             |           |           | $xxx.xx     |
| + 20% buffer |          |           |           | $xxx.xx     |

The 20% buffer is mandatory on all estimates. A budget ceiling must be stated.

7.3 Cost Sensitivity Analysis

Identify the top 3 cost drivers and state what happens if usage doubles.

7.4 Budget Alerts Are Mandatory
  • 80% of budget — Warning
  • 100% of budget — Critical
  • 90% of budget (forecasted) — Early warning

Alert configuration must be included as runnable CLI or IaC code — not just described.

8.1 Make a Decision — No Contradictions
OptionWhen to Use
Public PaaSDefault — no compliance requirement for private connectivity
Private (VNet + PE)Compliance requires private connectivity (SOC 2, HIPAA, enterprise policy)
HybridOn-premises connectivity required — specify ExpressRoute or VPN Gateway
8.2 Subnet Specifications (Private)
| Subnet Name | CIDR | Purpose | Min Size | NSG |

CIDRs must be non-overlapping, sized with 30% headroom for growth.

8.3 NSG Rules Must Be Complete

Every NSG must include a full rule table: Priority, Name, Direction, Protocol, Source, Destination, Port, Action. Default deny-all rules must be explicit.

8.4 Private Endpoints Must Be Named

Every PE: resource name, target service, DNS zone, NIC name, subnet placement.

8.5 DNS Must Be Explicit

Specify: Azure-managed public DNS, Private DNS Zones (list each), custom DNS servers (IPs), or conditional forwarders.

9.1 Rollback Strategy Required
LayerRollback Mechanism
Infrastructure (IaC)Re-deploy previous Git tag; cancel in-flight deployment
ApplicationSlot swap, revision rollback, or artifact re-deploy
DataPoint-in-time restore or migration rollback script
Configuration / SecretsKey Vault version rollback

A rollback decision tree must be included showing which lever to pull and in what order.

9.2 – 9.4 Pipeline Gates
  • Validate gate: Every CI/CD pipeline must include a validate/what-if step before any deployment
  • Production approval: Named human approver via environment protection rule — no auto-deploy to prod on push
  • Tag on deploy: Successful production deployments must create a Git tag: prod-YYYYMMDD-HHMMSS
9.5 Pipeline Secrets

All secrets must be listed with: name, description, location. OIDC preferred over long-lived credentials.

9.6 Pre-Deployment Checklist
[ ] Subscription / account access verified
[ ] Required IAM roles / permissions confirmed
[ ] Quota pre-checked for all resource types
[ ] IaC validated (what-if / plan) against target environment
[ ] Rollback target identified and tagged
[ ] On-call contact identified for deployment window
[ ] Budget alert configured
[ ] Monitoring dashboards ready

10.3 Acceptance Criteria Standards

Every AC must be: testable (a QA engineer can write a test without asking the author), binary (pass or fail), and assigned (maps to a feature or requirement ID).

Bad: "The system should be fast."
Good: AC-07 — The detail panel must render within 2s for a cached SKU at P95 under normal load.

11.1 Update When Systems Change

A spec that does not reflect the current system is worse than no spec. Update the spec before or simultaneously with the code change. Bump minor version + update date.

11.2 Deprecation Policy
  1. Set status to Deprecated
  2. Add notice: "This document is deprecated. See [replacement doc ID]."
  3. Retain for audit trail — do not delete
11.3 Review Cadence
Spec TypeReview Cadence
FunctionalBefore each sprint that touches the feature
DeploymentBefore each production deployment
ArchitectureQuarterly or when a major component changes
SecurityEvery 6 months or after any security incident
11.4 Spec Ownership

Every spec must have a named owner. When an owner leaves, ownership must be reassigned within 5 business days.
