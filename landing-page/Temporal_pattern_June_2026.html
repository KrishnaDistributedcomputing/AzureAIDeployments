Azure Landing Zone foundation

Azure Landing Zone becomes the foundation layer for the workflow platform. It establishes the subscription model, management group hierarchy, policy baseline, identity controls, network topology, observability boundary, and governance model before the API Layer, Temporal, AKS, PostgreSQL, HorizonDB, or AI workflow activities are deployed.

Strategy: treat Azure Landing Zone as the platform contract. The application team owns API contracts, workflow logic, and bounded AI behavior; the platform team owns reusable controls for identity, policy, network, security posture, logging, naming, tagging, and subscription vending.

Platform landing zone

Centralized shared services and guardrails used by all workload environments: management groups, policies, identity patterns, connectivity, security monitoring, platform logs, and standard deployment controls.

Shared servicesGovernanceConnectivity

Application landing zones

Workload subscriptions or resource groups that host the API Layer, AKS, Temporal workers, PostgreSQL, Key Vault, messaging, AI model endpoint integrations, and observability resources for each environment.

Workload-ownedEnvironment-specificTagged ownership

Landing zone pattern decisions

Design area Recommended pattern Why it matters for durable intelligent workflows
Management groups Separate platform, non-production, production, and sandbox management scopes. Lets policy, access, and governance controls vary by environment without changing application code.
Subscription vending Create repeatable workload subscriptions for sandbox, non-production, and production. Prevents every workflow team from inventing its own resource boundaries.
Policy baseline Require tags, allowed regions, diagnostic settings, private access where required, and approved service configurations. Stops uncontrolled resource drift, unapproved exposure, and inconsistent telemetry before production.
Identity baseline Use Entra ID, managed identities, workload identity for AKS pods, and role assignments by service boundary. Allows Temporal workers and AI activity services to access dependencies without embedded secrets.
Network topology Start with simple private networking; use hub-and-spoke or Virtual WAN only when shared connectivity and routing justify it. Keeps the early platform simple while leaving a path to enterprise connectivity.
Observability baseline Standardize Log Analytics workspaces, Application Insights, diagnostic settings, retention, and data collection rules. Workflow replay, AI decisions, retries, token usage and failure analysis need consistent telemetry.
Resource governance Use usage guardrails, alerts, tags, resource locks for critical resources, and environment expiration policy for sandboxes. Prevents idle clusters, verbose logs, and unmanaged model-call volume from becoming operational risk.

Business value across the solution stack

The architecture should be presented as a business acceleration pattern, not only a technology pattern. Each layer exists to reduce implementation friction, make customer-hosted and SaaS-hosted deployments more repeatable, keep operational data close to the workflow, and make AI useful inside real business processes.

Business thesis: start with the smallest reliable platform that can support regulated deployments, workflow-heavy SaaS, customer integrations, Postgres-first data ownership, and measurable AI improvement. Add sophistication only when the business process proves the need.

Temporal advantages for the customer pattern

Temporal is the preferred teaching and implementation pattern for workflow-heavy Azure solutions because it gives teams a durable, portable, code-first foundation for business processes. It should be presented as the recommended default for complex workflows, not as a mandatory product choice for every workload.

Customer-aligned position: teach a Kubernetes-ready foundation, keep the workflow engine explicit, and add intelligence inside Temporal activities as bounded decision or action steps. This keeps the architecture simple enough to teach while preserving room for regulated deployments, SaaS deployments, and future evaluation-driven improvement.

API Layer for component integration

The API Layer is the controlled integration surface between customer systems, SaaS applications, Temporal workflows, AKS services, PostgreSQL, bounded AI activity services, and observability. It keeps external systems from directly invoking workers or reaching into operational data stores.

Strategy: keep the API Layer thin and governed. It should authenticate, authorize, validate, route, apply policy, emit telemetry, and translate requests into workflow commands or domain-service calls. Temporal owns long-running process state; PostgreSQL owns operational data; workers own business execution.

North-south API boundary

Use Azure API Management or a governed API gateway pattern to expose workflow start, status, callback, approval, and domain APIs to customer systems, partners, internal applications, and SaaS tenants.

OpenAPI-firstVersioned contractsPolicy-driven

API Layer guardrails

  • OpenAPI-first: define request and response contracts before wiring workflows to external systems.
  • Explicit versioning: preserve older contract versions until dependent customers or systems have migrated.
  • Idempotency: every workflow-starting API should tolerate retries without duplicating business processes.
  • Correlation: pass a correlation ID from API entry through Temporal, worker logs, model calls, PostgreSQL writes, and observability.
  • Least privilege: API services should call only the workflow commands and domain services they require.
  • Private integration path: use private networking, private endpoints, or self-hosted gateway patterns where customer or regulated deployments require the API surface to remain inside controlled network boundaries.

Architecture patterns

P1. Learning / sandbox pattern

Best for teachingLow riskLow volume

A minimal end-to-end environment for proving the workflow-first pattern without committing to production-grade availability.

Topology

  • Azure Landing Zone LZ1 governance-only foundation.
  • Small AKS cluster configured for sandbox use.
  • Small Linux node pool, usually 1–2 nodes.
  • Temporal server, UI, and workers in one namespace.
  • Simple workflow command API for starting and querying demo workflows.
  • One PostgreSQL Flexible Server with separate databases/schemas.
  • ACR, Key Vault, and Azure Monitor with strict log limits.

Bounded AI design

  • Use a small model for extraction, routing, and summarization.
  • Store prompt version and output schema with each workflow decision.
  • Disable autonomous side effects; require explicit workflow action code.
Business value: gives teams a low-risk place to learn durable workflows, API contracts, Postgres-first data, and bounded AI steps before production commitments.

P2. Production MVP pattern

Default production startBounded AIOperationally simple

A practical first production deployment for one or two durable intelligent workflows.

Topology

  • Azure Landing Zone LZ2 production foundation.
  • Production AKS cluster management.
  • Three-node system/application pool, with autoscaling boundaries.
  • Temporal server separated from workflow workers by namespace and RBAC.
  • API Layer for workflow start/status/callback contracts and AKS ingress routing.
  • Production PostgreSQL configuration for persistence and app data, preferably separated by database.
  • ACR, Service Bus, Event Grid, Key Vault, and Application Insights.

Bounded AI design

  • Each AI activity has timeout, retry policy, schema validation, and fallback path.
  • Human approval is required for irreversible business actions.
  • Usage is tracked by workflow type, tenant/customer, model, and prompt version.
Business value: converts the learning pattern into a repeatable production workflow system with governed integrations, auditable AI decisions, and operational ownership.

P3. HA production pattern

Business criticalZone-awarePrivate networking

A production-ready baseline for customer-facing workflows where downtime, recovery, and auditability matter.

Topology

  • Azure Landing Zone LZ3 enterprise foundation.
  • Zone-aware AKS node pools.
  • Separate node pools for Temporal server, workflow workers, and burstable task workers.
  • Hardened API Layer with private integration path where required.
  • PostgreSQL high-availability posture appropriate to the deployment environment.
  • Private endpoints for database, registry, and Key Vault where required.
  • Separate staging and production environments.

Operational rules

  • Backups tested monthly with restore drills.
  • Workflow replay/versioning strategy defined before production changes.
  • SLOs cover workflow latency, retry count, completion rate, model-call error rate, and queue age.
Business value: protects customer-facing and business-critical processes with resilience, private access options, restore practice, and clear service ownership.

P4. High-throughput workflow + Postgres scale pattern

Scale gateDatabase boundHorizonDB evaluation

Used when the bottleneck is workflow concurrency, database throughput, vector lookup performance, or operational data scale.

Topology

  • Azure Landing Zone LZ3 with production subscription, private routing, and centralized monitoring patterns.
  • Dedicated AKS pools by workload class: workflow server, online workers, batch/eval workers.
  • API contracts partitioned by workflow domain, tenant, and integration type.
  • PostgreSQL read replicas, partitioning, connection pooling, and separate eval/history stores.
  • Vector tables remain close to operational data unless benchmark data proves otherwise.
  • HorizonDB proof-of-value path for scale-out PostgreSQL-compatible workloads.

Scale controls

  • Queue partitions by tenant, workflow type, or priority.
  • Rate-limit model calls at the activity boundary.
  • Use replay-safe versioning and shard large workflows into child workflows.
Business value: lets the platform grow only where telemetry proves the bottleneck, preserving simplicity while supporting higher workflow volume and data intensity.

P5. Trace/evaluation readiness pattern

Quality loopTuning prerequisiteEvidence first

Adds the data discipline needed before advanced optimization or tuning should be considered.

Topology

  • Persist workflow outcome, model input hash, output, validation result, and human correction.
  • Store eval datasets separately from production workflow tables.
  • Run scheduled eval workers on AKS with controlled model-call limits.

Readiness gates

  • At least one stable eval set per workflow type.
  • Clear success/failure rubric and owner.
  • Feedback labels collected from real workflow outcomes.
Business value: turns workflow execution into a measurable improvement loop by capturing outcomes, corrections, labels, and evaluation signals.

P6. API Layer integration pattern

API-firstCustomer integrationsWorkflow commands

A stable API boundary for starting workflows, querying state, receiving callbacks, and integrating with customer systems without exposing Temporal internals or operational databases.

Topology

  • API Management or governed API gateway pattern as the north-south boundary.
  • AKS ingress routes to workflow command services and domain APIs.
  • Workflow command service maps HTTP requests to Temporal starts, signals, and queries.
  • Service Bus and Event Grid bridge synchronous APIs to asynchronous workflow execution.
  • Private integration path for customer-hosted, regulated, or network-isolated deployments.

Contract rules

  • OpenAPI contracts for workflow commands, status, callbacks, approvals, and domain APIs.
  • Idempotency keys for workflow-starting APIs.
  • Correlation IDs propagated through API Layer, Temporal, workers, PostgreSQL, model calls, and logs.
  • Tenant/customer authorization before workflow start, signal, query, or callback handling.
  • No direct database access from external systems.
Business value: shortens customer and partner integration work by providing stable contracts, callback rules, idempotency, and a safe boundary around workflow execution.

Spec-Driven Development: Azure Backup Demo Environment

Document version: 1.0
Target audience: Cloud engineering, customer engineering, solution architecture, sales engineering, demo presenters
Primary objective: Build a repeatable Azure Backup demo environment that can show backup configuration, policy design, restore workflows, operational monitoring, and ransomware-resilience controls.

1. Executive Summary

This specification defines a demo environment for Azure Backup. The environment is designed to demonstrate how an enterprise can protect Azure workloads using native Azure backup services, validate recovery scenarios, and show operational governance around backup security.

  • How Azure Backup protects virtual machines, files, disks, databases, and newer Azure data sources.
  • How backup vaults and Recovery Services vaults are selected based on workload type.
  • How backup policies define frequency, retention, and recovery objectives.
  • How recovery is performed through VM restore, file-level restore, disk restore, and workload-specific restore.
  • How soft delete, immutability, monitoring, alerts, RBAC, and private access patterns strengthen backup resilience.

2. Demo Goals

2.1 Primary Goals

  1. Protect representative Azure workloads.
  2. Use policy-driven backup configuration.
  3. Restore data from recovery points.
  4. Demonstrate backup isolation and recovery control.
  5. Show backup monitoring and operational alerts.
  6. Demonstrate security controls for accidental or malicious deletion scenarios.
  7. Provide a reusable environment that can be deployed, demonstrated, and cleaned up.

2.2 Secondary Goals

  1. Recovery Services vault versus Backup vault selection.
  2. Operational backup ownership and RBAC.
  3. Cost factors such as protected instances, retained data, storage redundancy, and retention.
  4. Enterprise patterns for landing zone integration.
  5. How backup fits into a broader business continuity and ransomware recovery strategy.

10.2 Soft Delete

Soft delete must remain enabled for the demo. It protects backup data from accidental or malicious deletion by retaining deleted backup items for a configured period.

  1. Protect VM with Azure Backup.
  2. Stop protection and attempt to delete backup data.
  3. Show that deleted backup items are retained through soft delete.
  4. Explain how this helps recover from accidental or malicious deletion.

10.3 Immutable Vault

Immutable vault is optional. Use it only if the presenter understands cleanup implications. Show the setting and explain its purpose, but do not lock immutability in a disposable demo unless the cleanup process has been tested.

10.4 Network Security

Simple demo

  • Public Azure control plane access is acceptable.
  • No public RDP/SSH inbound access unless explicitly required.

Advanced demo

  • Use Azure Bastion for VM access.
  • Use private endpoints where supported.
  • Restrict storage account network access.
  • Use Azure Policy to deny public IP creation.

12. Demo User Stories

Story 1: Protect an Azure VM

As a backup administrator, I want to configure backup for an Azure VM, so that I can recover the VM or its data if it is deleted, corrupted, or changed incorrectly.

  • VM appears as a protected item in the Recovery Services vault.
  • Backup policy is assigned.
  • Initial backup job can be triggered.
  • Recovery point is created.
  • Backup job status is visible.

Story 2: Restore a VM

As a restore operator, I want to restore a VM from a recovery point, so that I can recover from a workload failure.

  • A recovery point is selected.
  • Restore option is shown.
  • Restore to new VM or restore disk is demonstrated.
  • Restored resource is clearly tagged as demo restore output.
  • Presenter explains when to use full VM restore versus disk restore.

Story 3: Restore a File

As a workload owner, I want to restore a deleted file from backup, so that I can recover from accidental deletion without restoring the full VM.

  • Sample file is created.
  • Backup recovery point exists.
  • File is deleted or modified.
  • File-level restore workflow is demonstrated.
  • Restored file is validated.

Story 4: Protect Azure File Share

As a platform operator, I want to back up an Azure file share, so that shared application or user data can be recovered.

  • Storage account and file share are deployed.
  • File share contains sample files.
  • Backup is configured.
  • Recovery point is created.
  • Restore workflow is demonstrated.

Story 5: Demonstrate Backup Security

As a security stakeholder, I want to understand how backup data is protected from deletion, so that I can assess ransomware recovery posture.

  • Soft delete setting is shown.
  • Deleted backup item behavior is explained.
  • Immutable vault concept is explained.
  • RBAC separation is explained.
  • Backup alerting is shown.

Story 6: Monitor Backup Health

As an operations team member, I want to monitor backup job status and failures, so that I can quickly identify backup issues.

  • Backup jobs are visible in the vault.
  • Alerts or diagnostic settings are configured.
  • Logs are queryable in Log Analytics.
  • Failed job query is available.
  • Presenter explains operational handoff.

20. Cost Control Requirements

Cost-control decisions

  • Use small VM sizes.
  • Use locally redundant storage unless cross-region restore is required.
  • Use short retention.
  • Delete restored resources after demo.
  • Avoid production-scale retention policies.
  • Avoid enabling immutable locked settings unless required.
  • Use auto-delete tags.
  • Run cleanup script after the demo.

Cost drivers to explain

  1. Number of protected instances.
  2. Backup storage consumed.
  3. Retention duration.
  4. Storage redundancy.
  5. Snapshot retention.
  6. Restore test resources.
  7. Monitoring/log ingestion.

22. Demo Script

Opening

"Today we are demonstrating how Azure Backup can protect core Azure workloads using policy-driven backup, centralized vault management, restore workflows, and built-in protection controls such as soft delete and optional immutability."

Explain Vaults

"Azure Backup uses different vault types depending on workload. Recovery Services vaults are commonly used for Azure VM and workload backup scenarios. Backup vaults are used for certain newer Azure data source backup scenarios. The demo shows the distinction so teams understand which vault pattern to use."

VM Backup

"This VM represents a typical application server. It has a backup policy assigned, a recovery point available, and can be restored either as a full VM, a disk, or through file-level recovery depending on the recovery scenario."

Azure Files Backup

"This file share represents shared application or business data. Azure Backup can protect the file share and allow restore of deleted or modified files."

Security

"Backup is not only about copies of data. The security model matters. Soft delete and vault immutability help reduce the risk of accidental or malicious deletion of backup data. RBAC separates who can configure backup from who can perform restore or delete operations."

Close

"The key takeaway is that Azure Backup provides native, policy-driven protection for Azure workloads with operational visibility, restore validation, and security controls that are important for business continuity and ransomware recovery planning."

27. Summary

This specification creates a structured, repeatable, and explainable Azure Backup demo environment. It is intentionally designed to be small enough for a controlled demo while still showing enterprise-relevant capabilities: vault selection, policy-driven backup, restore validation, monitoring, RBAC, soft delete, and optional immutability.

Demo posture: The environment should be treated as a disposable technical demo, not a production backup architecture. Production implementation requires workload-specific RPO/RTO design, retention policy definition, compliance review, security review, cost modeling, and restore testing at operational scale.
