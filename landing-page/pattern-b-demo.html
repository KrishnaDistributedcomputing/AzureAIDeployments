Pattern B: Ransomware Resilience

Role-Based Access Control, Soft Delete, & Immutable Vault Lock

Security-First Pattern

Pattern Overview

Pattern B demonstrates how to protect Azure workloads from ransomware attacks using role-based access control (RBAC) separation, soft delete recovery windows, and immutable vault locks.

🎯 Target Audience Security stakeholders, CISOs, backup administrators, business continuity teams
⏱️ Deployment Time 25-30 minutes automated deployment

Architecture

Pattern B separates workload (application VMs, storage) from protection infrastructure (vaults, monitoring). This isolation prevents ransomware from disabling backups.

Key Security Features

🔐 Soft Delete (14-day recovery)

Even if ransomware deletes backups, they remain recoverable for 14 days. This provides a critical recovery window for incident response.

Ransomware Protection: Attacker deletes backups → Backups enter soft-delete state → Undelete within 14 days
🔒 Immutable Vault Lock

Once enabled, the vault cannot be deleted or modified. This provides permanent protection against ransomware and insider threats.

Permanent Protection: Lock prevents vault deletion, policy modification, and downgrade of retention
👥 RBAC Separation

Different roles have different permissions. The operator who triggers backups cannot modify vault settings, preventing insider attacks.

Backup Operator Admin Restore Op Audit Reader

📊 Audit Trail

All vault operations logged and queryable in Log Analytics. Track who accessed backups, when, and from where.

Compliance Proof: Immutable audit log demonstrates backup integrity for regulatory audits

Role-Based Access Control Model

Pattern B implements 4 security roles with clear separation of duties. This prevents any single person from both backing up and disabling protections.

Role Permissions Use Case Risk Level
Backup Operator Trigger backups, initiate restores (no config) Daily backup operations, incident response LOW
Backup Administrator Full vault management, policy creation, retention Vault design, policy tuning, retention decisions HIGH
Restore Operator Restore workflows only, view recovery points File recovery, incident response, testing MEDIUM
Audit Reader Read-only access to vault, logs, policies SOC teams, compliance audits, CIO office LOW
Key Principle: The account that triggers backups cannot modify vault settings. This prevents ransomware from disabling backups after compromising a machine.

Demo Scenarios

Scenario 1: RBAC Role Separation (5 minutes)

Objective: Show that RBAC prevents unauthorized backup policy modifications

# Backup Operator CAN trigger restore but CANNOT modify policy az backup protection update-for-vm ... --as-user "backup-operator@company.com" # Result: Permission denied ✗ # Backup Administrator CAN modify policy az backup protection update-for-vm ... --as-user "backup-admin@company.com" # Result: Policy updated ✓

Talking Points:

  • "The operator who triggers backups cannot disable protections"
  • "This prevents ransomware from disabling backups after VM compromise"
  • "Different teams have different levels of trust"
Scenario 2: Soft Delete Recovery (Ransomware Simulation) (8 minutes)

Objective: Show 14-day recovery window for "deleted" backups

# 1. "Delete" a recovery point (soft delete, not permanent) az backup recoverypoint delete --vault-name ... --recovery-point-id ... # 2. Show it's in soft-delete state (still recoverable) az backup recoverypoint list --query "[].{state:state, deleteState:deleteState}" # 3. Undelete within 14-day window az backup recoverypoint undelete --vault-name ... --recovery-point-id ...

Talking Points:

  • "Even if attacker deletes our backups, we have 14 days to recover them"
  • "This 14-day window is the critical incident response period"
  • "Immutable lock ensures we can't 'accidentally' disable soft delete"
Scenario 3: Monitoring & Real-Time Alerts (6 minutes)

Objective: Show security alerts triggered on suspicious activity

# Alert 1: Backup job failures (3+ failures in 1 hour) # Alert 2: Vault deletion attempts # Alert 3: Soft delete operations # Alert 4: Unauthorized restore attempts # View alerts in real-time az monitor alert list --query "[?contains(name, 'backup')]"

Talking Points:

  • "Multiple failed backups often indicate ransomware"
  • "Real-time alerts enable faster incident response"
  • "SOC team notified immediately of suspicious activity"

Restore Workflows

Pattern B supports 3 restore strategies, each optimized for different scenarios:

📁 File-Level Restore (Fastest)

Typical Time: 3-5 minutes

Browse and restore individual files from a backup without restoring the entire disk.

Use Case: Accidental file deletion, ransomware infected specific files

💾 Disk Restore (Full Recovery)

Typical Time: 5-15 minutes

Restore entire encrypted data disk to alternate VM for testing or recovery.

Use Case: Complete disk corruption, full system recovery

🔓 Soft Delete Recovery (Ransomware Specific)

Typical Time: < 1 minute

Undelete backup recovery points that were deleted during ransomware attack.

Use Case: Ransomware deleted backups, need to recover from soft-deleted backup

Frequently Asked Questions

What's the difference between soft delete and immutable lock?

Soft Delete: Allows undeletion for 14 days. Can still be permanently deleted by administrator with permission.

Immutable Lock: Permanent protection. Vault cannot be deleted, modified, or downgraded. Requires Microsoft intervention to remove.

Together: Soft delete provides short-term ransomware recovery; immutable lock provides permanent protection.

Can ransomware still attack if soft delete is enabled?

Yes, ransomware can still try to delete backups. But with soft delete, even if deleted, backups remain in a soft-delete state for 14 days and can be recovered.

This 14-day window is the critical period for incident response. Immutable lock adds permanent protection.

Why separate Workload RG and Protection RG?

Separation provides defense in depth. Even if ransomware compromises the application VM, it cannot access the backup vault (in a different RG with different permissions).

What happens if I delete a vault by mistake?

With immutable lock enabled, deletion is blocked. You'll get an error: "Vault is locked and cannot be deleted." To permanently remove the vault, you must first disable immutable lock (requires Microsoft support in some cases).

Can I restore individual files without restoring the entire disk?

Yes! File-level restore (Scenario 1) allows browsing and restoring individual files. This is the fastest and most granular recovery option.

How long does it take to restore from a backup?

Restore times vary: File-level restores: 3-5 minutes, Disk restores: 5-15 minutes, Soft delete recovery: < 1 minute

Is this pattern compliant with HIPAA/PCI-DSS/SOX?

Pattern B includes features that support compliance (RBAC separation, audit trail, immutable lock), but full compliance requires additional controls like network isolation, encryption, and documentation. Use Pattern D (Enterprise Landing Zone) for full CAF-aligned compliance setup.
