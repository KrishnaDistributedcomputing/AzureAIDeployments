Overview

Core Objective

Protect Azure workloads with policy-driven backup, validated restore, centralized monitoring, least privilege, soft delete, optional immutability, and clean teardown for demo environments.

Recovery Services vault
Core pattern for VM, Files, SQL
Backup vault
Modern pattern for Disk, AKS, Blob
Log Analytics
Operational monitoring & alerts
RBAC + alerts
Security & least privilege

What this page defines: This page converts Azure Backup planning into a spec-driven deployment pattern. Each use case identifies the workload, vault type, policy shape, restore workflow, security controls, monitoring evidence, and cleanup requirements before infrastructure is deployed.

The goal is to support both a low-cost demo environment and production conversation starters for enterprise backup design. Demo defaults favor small resources, short retention, locally redundant storage, pre-staged recovery points, and tested cleanup. Production decisions should be driven by RPO, RTO, compliance, data classification, regional availability, restore testing, and cost modeling.

Spec sections: Scenario, Vault choice, Policy, Restore path, Security, Monitoring, Cost controls, Cleanup

Use Case Catalog

Azure Backup use cases and deployment intent

Azure VM Backup

Protect Windows or Linux VMs with crash-consistent or app-consistent recovery points. Demonstrate VM restore, disk restore, and file-level recovery.

Recovery Services vault Must-have demo
Azure Files Backup

Protect Azure file shares for accidental deletion, ransomware simulation, and point-in-time file or share restore demos.

Recovery Services vault File restore
Azure Disk Backup

Use snapshot-based managed disk protection for application-owned disks and newer Backup vault workflows.

Backup vault Optional demo
AKS Backup

Protect Kubernetes cluster state and persistent volumes where AKS Backup is available and aligned to workload recovery needs.

Backup vault Extension pattern
Azure Blob Operational Backup

Demonstrate operational protection for containers with soft delete, versioning, point-in-time restore, and policy governance.

Backup vault Data protection
Azure Database for PostgreSQL

Use workload-native backup capability and, where supported, vaulted backup patterns for longer retention and centralized governance.

Backup vault Where supported
SQL Server in Azure VM

Protect SQL databases inside Azure VMs with workload-aware policies, log backup, and database-level restore workflows.

Recovery Services vault Production extension

Enterprise workload protection for SAP HANA databases with validated backup registration, policies, and restore drills.

Recovery Services vault Advanced extension
On-Premises Servers

Use MARS, MABS, or DPM for Windows files, folders, system state, or workloads that must back up to Azure from outside Azure.

Recovery Services vault Hybrid

Vault Selection

Recovery Services vault vs Backup vault

Decision Area Recovery Services Vault Backup Vault
Typical Workloads Azure VM, Azure Files, SQL in Azure VM, SAP HANA in Azure VM, MARS/MABS/DPM scenarios Azure Disk Backup, AKS Backup, Blob operational backup, newer data sources
Demo Priority Required for core VM and Azure Files demo Recommended for disk or newer workload demo
Security Controls Soft delete, enhanced security, RBAC, alerts, diagnostic logs, immutability RBAC, diagnostic logs, datastore settings, immutability features
Cost Posture Protected instances, retained data, redundancy, snapshot retention Snapshot/vaulted backup retention and datastore choice
Demo Tip: Start with Recovery Services Vault for VM and Azure Files demo, then explain Backup vault as the newer pattern. Do not lock immutability in disposable demos unless cleanup is tested.

Spec-Driven Deployment Patterns

Patterns for different Azure Backup use cases

Pattern A: Core Demo

Best for sales engineering, customer engineering, and solution architecture demos.

  • One Windows or Linux VM with data disk
  • One storage account with Azure file share
  • Recovery Services vault with daily policies
  • Log Analytics workspace & diagnostics
  • Soft delete enabled; immutability explained

Acceptance: VM and file share protected, recovery point exists, one restore path demonstrated, cleanup tested.

Deployment .md SPEC.md Demo

Pattern B: Ransomware Resilience

Best for security and business continuity stakeholders.

  • RBAC separation (backup admin, restore operator)
  • Soft delete demonstration for deleted items
  • Immutable vault guidance
  • Azure Monitor alerts for job failure
  • Pre-staged recovery point

Acceptance: Presenter explains deletion protection, role separation, restore control, and alert path.

Deployment .md SPEC.md Demo

Pattern C: Modern Workloads

Best for platform teams evaluating Backup vault patterns.

  • Backup vault deployed with diagnostics
  • Optional managed disk backup policy
  • Optional AKS backup extension
  • Optional Blob operational backup
  • Clear vault distinctions

Acceptance: Team understands why Backup vault exists and which workloads use it.

Deployment .md SPEC.md Demo

Pattern D: Enterprise Landing Zone

Best for production-readiness conversations.

  • Separate resource groups (workload, protection, ops)
  • Policy-driven tags & diagnostic settings
  • Private access with Bastion & private endpoints
  • Cost controls through retention decisions
  • Runbooks for deployment & handoff

Acceptance: Backup design mapped to CAF governance, security, operations, and costs.

Runbook

Demo flow from deploy to restore

1. Deploy the Environment

Create workload, protection, and operations resource groups; deploy VM, storage account, vaults, policies, and diagnostics.

2. Seed Demo Data

Create sample VM files and Azure file share folders such as finance, operations, and customer documents.

3. Enable Protection

Assign policies, enable backup for the VM and file share, and optionally configure disk or modern data-source backup.

4. Trigger or Review Backup

Run initial backup or use a pre-staged recovery point when the live job would take too long.

5. Demonstrate Restore

Show full VM restore, disk restore, file-level recovery, or Azure Files item restore depending on audience.

6. Show Security Controls

Review soft delete, immutability guidance, RBAC separation, and no-public-access posture.

7. Show Monitoring

Open backup jobs, alerts, protected items, recovery points, and Log Analytics queries.

8. Clean Up

Remove restored resources, stop protection where appropriate, validate soft-delete implications, and delete demo resource groups.

Monitoring

Operational monitoring queries

Backup Jobs

AzureDiagnostics | where Category has "AzureBackup" | project TimeGenerated, Resource, Category, OperationName, ResultType, ResultDescription | order by TimeGenerated desc

Failed Operations

AzureDiagnostics | where Category has "AzureBackup" | where ResultType !in ("Success", "Succeeded") | project TimeGenerated, Resource, OperationName, ResultType, ResultDescription | order by TimeGenerated desc

Vault Activity

AzureActivity | where ResourceProviderValue =~ "Microsoft.RecoveryServices" | project TimeGenerated, OperationNameValue, ActivityStatusValue, Caller, ResourceGroup, Resource | order by TimeGenerated desc

Infrastructure as Code

Implemented spec-driven deployment package

A dedicated deployable package is available in the repository at azure-backup-demo-spec/. It contains the Bicep entry point, focused modules, scripts, docs, and KQL queries needed to deploy and demonstrate the Azure Backup environment.

azure-backup-demo-spec/ README.md SPEC.md /infra main.bicep main.demo.bicepparam /modules recovery-services-vault.bicep backup-vault.bicep vm.bicep storage.bicep monitoring.bicep rbac.bicep /scripts deploy.ps1 seed-demo-data.ps1 trigger-backup.ps1 validate-backup.ps1 cleanup.ps1 /docs architecture.md demo-runbook.md presenter-notes.md test-plan.md /queries backup-jobs.kql failed-backups.kql vault-activity.kql

Deploy Command

./scripts/deploy.ps1 ` -Location canadacentral ` -EnvironmentName backup-demo ` -Owner krishna ` -AdminPassword (Read-Host -AsSecureString "VM admin password")

Required Parameters

environmentName: backup-demo location: canadacentral owner: krishna deployWindowsVm: true deployLinuxVm: false deployAzureFiles: true deployDiskBackup: false enableSoftDelete: true enableImmutableVault: false retentionDays: 7 storageRedundancy: LRS
Cleanup Warning: Do not lock immutable vault settings in a disposable demo unless the teardown process has been tested with that setting enabled.

Deploy to Azure

How to deploy the spec-driven patterns

1. Prepare the Subscription

  • Select a subscription where you can create resource groups, managed identities, vaults, virtual machines, storage accounts, and Log Analytics workspaces.
  • Install Azure CLI, PowerShell 7, and Bicep support through Azure CLI.
  • Confirm provider registration for Microsoft.RecoveryServices, Microsoft.DataProtection, Microsoft.Compute, Microsoft.Storage, Microsoft.Network, and Microsoft.OperationalInsights.
  • Use a short-lived demo owner, cost center, and deletion date before deploying.

2. Choose the Pattern Switch Set

Pattern Recommended Switches
A Core demo deployWindowsVm=true, deployAzureFiles=true, deployDiskBackup=false, enableImmutableVault=false
B Ransomware resilience deployWindowsVm=true, deployAzureFiles=true, enableSoftDelete=true, enableImmutableVault=false
C Modern workloads deployBackupVault=true, deployDiskBackup=true, optional AKS or Blob backup extension
D Enterprise landing zone Same as A, B, or C plus RBAC principal IDs, required tags, diagnostic settings, policy assignments

3. Run the Deployment

cd azure-backup-demo-spec az login az account set --subscription "<subscription-name-or-id>" az provider register --namespace Microsoft.RecoveryServices az provider register --namespace Microsoft.DataProtection $env:AZURE_BACKUP_DEMO_ADMIN_PASSWORD = "<strong-password>" ./scripts/deploy.ps1 ` -Location canadacentral ` -EnvironmentName backup-demo ` -Owner <owner-name> ` -AdminPassword (ConvertTo-SecureString $env:AZURE_BACKUP_DEMO_ADMIN_PASSWORD -AsPlainText -Force)

For a shared environment, replace the plain environment variable step with your secret store or a secure PowerShell prompt.

4. Validate and Operate

./scripts/seed-demo-data.ps1 -EnvironmentName backup-demo ./scripts/trigger-backup.ps1 -EnvironmentName backup-demo ./scripts/validate-backup.ps1 -EnvironmentName backup-demo
  • Confirm protected items appear in the Recovery Services vault or Backup vault.
  • Confirm a recovery point exists before the live presentation.
  • Open Log Analytics and run the KQL queries from queries/.
  • Complete cleanup after the demo with ./scripts/cleanup.ps1 -EnvironmentName backup-demo.
Production note: For Pattern D, do not deploy directly from demo defaults. Convert the selected pattern into a reviewed landing-zone spec with approved naming, tags, policy assignments, RBAC groups, RPO/RTO, retention, private access, monitoring ownership, and cost-management approval.

Definition of Done

The deployment is ready when:

  • Environment deploys from IaC
  • VM and Azure Files backup are protected
  • Recovery points exist
  • At least one restore is validated
  • Backup jobs and alerts are visible
  • Soft delete behavior is explained
  • Cleanup is tested
  • Presenter can complete the demo in 30 to 45 minutes
