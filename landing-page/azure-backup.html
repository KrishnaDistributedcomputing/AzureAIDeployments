Pattern A: Core Demo Environment

Azure Backup Spec-Driven Deployment for Sales & Customer Engineering

Compact demo with VM + Azure Files protection, 7-day retention, Log Analytics diagnostics, and restore workflows.

Spec ID: AZB-SDD-PATTERN-A-CORE-DEMO Deployment: ~20 minutes
Contents
Objective Use Cases Resources Tags Switches Deploy Seed & Protect Restore Demo Acceptance Cleanup

Objective

Deploy a compact, time-limited demo environment for Azure Backup that protects:

  • One Windows VM — Operating system and data disks with daily snapshots
  • One Azure file share — Business-critical data with granular item-level restore
  • Recovery Services vault — Central backup management with 7-day retention
  • Log Analytics diagnostics — Backup job monitoring and performance analytics
  • Restore workflows — VM file recovery, disk restore, and file share item restore

Primary Use Cases

Sales Engineering

Demonstrate backup capabilities to prospects without long-term infrastructure costs.

Customer Engineering

Validate backup strategy and retention policies before production rollout.

Solution Architecture

Test backup patterns, restore procedures, and disaster recovery workflows.

Executive Demo

Show tangible business value: data protection, compliance, quick recovery.

Deployment Steps

1. Prerequisites & Provider Registration

# Login and set subscription az login az account set --subscription "<subscription-name-or-id>" # Register required resource providers az provider register --namespace Microsoft.RecoveryServices az provider register --namespace Microsoft.DataProtection az provider register --namespace Microsoft.Compute az provider register --namespace Microsoft.Storage az provider register --namespace Microsoft.Network az provider register --namespace Microsoft.OperationalInsights # Verify registration (should show State: Registered) az provider show --namespace Microsoft.RecoveryServices --query "registrationState"

2. Deploy Infrastructure with Bicep

cd azure-backup-demo-spec # Capture admin password securely $adminPassword = Read-Host -AsSecureString "Enter VM admin password (min 12 chars, complex)" # Deploy all resources ./scripts/deploy.ps1 ` -Location canadacentral ` -EnvironmentName backup-demo ` -Owner "John Smith" ` -RegionCode cc ` -AutoDeleteAfter "2026-07-22" ` -AdminUsername azureadmin ` -AdminPassword $adminPassword # Expected output: 3 resource groups, ~10 resources, ~10 minutes

3. Verify Deployment

# List created resource groups az group list --query "[?contains(name, 'backup-demo')].{Name:name, Location:location}" -o table # List resources in core group az resource list --resource-group rg-backup-demo-core -o table # Verify vault creation az backup vault list --resource-group rg-backup-demo-protection -o table

Seed Data, Protect, & Validate

Step 1: Create Demo Data

Populate VM and file share with sample data for backup demonstration.

$CoreResourceGroupName = "rg-backup-demo-core" $StorageAccountName = az storage account list ` --resource-group $CoreResourceGroupName ` --query "[?starts_with(name, 'stbackupdemo')].name | [0]" -o tsv $FileShareName = "demo-share" ./scripts/seed-demo-data.ps1 ` -ResourceGroupName $CoreResourceGroupName ` -StorageAccountName $StorageAccountName ` -FileShareName $FileShareName # Creates: # - VM: C:\DemoData\*.txt (sample files) # - File Share: demo-share/reports/* (sample PDFs, CSVs)

Step 2: Trigger Initial Backups

Execute backup jobs for both VM and file share.

$CoreResourceGroupName = "rg-backup-demo-core" $ProtectionResourceGroupName = "rg-backup-demo-protection" $VaultName = "rsv-backup-demo-cc" $VmName = "vm-win-backup-demo-01" $FileShareName = "demo-share" $StorageAccountName = az storage account list ` --resource-group $CoreResourceGroupName ` --query "[?starts_with(name, 'stbackupdemo')].name | [0]" -o tsv ./scripts/trigger-backup.ps1 ` -CoreResourceGroupName $CoreResourceGroupName ` -ProtectionResourceGroupName $ProtectionResourceGroupName ` -VaultName $VaultName ` -VmName $VmName ` -StorageAccountName $StorageAccountName ` -FileShareName $FileShareName ` -RetainUntil (Get-Date).AddDays(7) # Expected: Backup jobs queued, processing ~3-5 minutes for first backup

Step 3: Validate Backup Completion

Confirm recovery points exist and backup job metrics are visible.

./scripts/validate-backup.ps1 ` -ProtectionResourceGroupName rg-backup-demo-protection ` -VaultName rsv-backup-demo-cc # Checks: # ✓ VM has backup items # ✓ File share has backup items # ✓ Recovery points exist (last 7 days) # ✓ Backup jobs completed successfully # ✓ Log Analytics shows backup metrics

Restore Demo

Demonstrate one primary restore path to show backup value. Choose one scenario:

Option A: VM File Recovery (Easiest)

Scenario: User accidentally deleted a file from VM. Recover specific file from recovery point.

./scripts/restore-vm-files.ps1 ` -ProtectionResourceGroupName rg-backup-demo-protection ` -VaultName rsv-backup-demo-cc ` -VmName vm-win-backup-demo-01 ` -RecoveryPointDate (Get-Date).AddDays(-1) # Opens file picker to select files from recovery point # Downloads .iso with recoverable files for mounting

Option B: Restore Disks (Medium Complexity)

Scenario: VM corruption detected. Restore OS disk to new VM.

./scripts/restore-vm-disks.ps1 ` -ProtectionResourceGroupName rg-backup-demo-protection ` -VaultName rsv-backup-demo-cc ` -VmName vm-win-backup-demo-01 ` -RestoreResourceGroupName rg-restore-temp # Creates managed disks in rg-restore-temp # User can attach to new VM for verification

Option C: Azure Files Item Restore (Recommended)

Scenario: Business-critical report file was deleted. Restore from file share backup.

./scripts/restore-file-share-items.ps1 ` -ProtectionResourceGroupName rg-backup-demo-protection ` -VaultName rsv-backup-demo-cc ` -FileShareName demo-share ` -RecoveryPointDate (Get-Date).AddDays(-2) ` -FilePath "reports/Q2-Summary.pdf" # Restores file to original or alternate location # Item-level granularity shows powerful retention capability

Acceptance Criteria

Verify successful deployment before showcasing to stakeholders.

  • All 3 resource groups created with correct naming
  • VM deployed, provisioned, and accessible via Bastion/RDP
  • Azure file share created and mounted on VM
  • Recovery Services vault deployed with soft delete enabled
  • Backup policies (VM + Files) assigned and active
  • Demo data seeded on VM and file share (≥100MB total)
  • Initial backup job completed successfully
  • Recovery points visible in vault (≥1 per data source)
  • Backup metrics showing in Log Analytics
  • One restore path executed successfully
  • All resources tagged with required tags + AutoDeleteAfter date
  • Cost tracking shows monthly estimate (<$50/month)
  • Cleanup script tested and resources can be deleted in <5 minutes

Cleanup & Deprovisioning

Demo resources are temporary. Clean up after demo/evaluation period.

Manual Cleanup

# Option 1: Delete entire environment az group delete --name rg-backup-demo-core --yes --no-wait az group delete --name rg-backup-demo-protection --yes --no-wait az group delete --name rg-backup-demo-ops --yes --no-wait # Option 2: Guided cleanup script ./scripts/cleanup-demo.ps1 -Force # Expected: All resources deleted within 5 minutes

Automated Cleanup (Recommended)

Best Practice: Use AutoDeleteAfter tag with Azure Policy lifecycle management to automatically delete resources on specified date. No manual intervention required.
# Azure Policy automatically evaluates AutoDeleteAfter tag # If today >= AutoDeleteAfter date, resource is marked for deletion # Optional 7-day grace period before permanent deletion # View policy-tagged deletions az resource list --tags AutoDeleteAfter=2026-07-22 -o table

Key Learnings & Best Practices

Soft Delete Protection

Enables recovery of deleted backup items for 14 days, critical for accidental deletions.

Retention Strategy

Demo uses 7-day retention. Production typically uses 30/60/365-day policies.

Backup Frequency

Daily backups recommended for VMs/files. Hourly granularity available for premium workloads.

Restore Testing

Test at least one restore path monthly. Backups are only valid if verified restorable.
